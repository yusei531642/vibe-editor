# judgment-policy クイック判定表

ワーカーの instructions に埋め込む圧縮版。迷ったらまずこれを見る。
詳細は `judgment-policy/SKILL.md` を参照（インストール済みの場合）。

## 基本原則

1. **シンプル志向** — 複雑化は最大の敵
2. **完成度最優先** — 暫定パッチ禁止、根本解決+テストまで
3. **第一原理思考** — 詰まったら原点に戻る
4. **勇気ある撤退** — 道がないなら退く判断も価値
5. **コードベース慣習尊重** — 既存パターン最優先

## 操作安全マトリクス（抜粋）

| 操作 | 動作 |
|------|------|
| 可逆+ローカル+低コスト | 聞かずに実行 |
| 可逆+共有資源 | 1行報告して実行 |
| 不可逆操作（force push, reset --hard等） | 確認 |
| main merge / 本番デプロイ | 必ず確認 |
| 破壊的DB migration | 確認 |
| 追加的DB migration | 聞かずに実行 |

## 失敗時のルール

- 同一仮説で2回失敗 → Codex相談
- 高リスク領域で1回失敗 → 即Codex相談
- lint/型エラー（自明） → 自力修正
- FLAKY（ネットワーク瞬断等） → リトライ

## 競合解決の優先順位

1. 明示的なタスク指示（ユーザー指示）
2. 実行中skillのhard gate
3. judgment-policy
4. 汎用autopilot系skill

## ユーザーに聞くべき場面（JP-99）

以下のいずれかに該当するときのみ:
- 競合ルールの優先順位が決められない
- 不可逆・高額・本番影響がある
- 要件解釈でユーザー価値が変わる
- どのJPセクションでも答えが出ない

「念のため聞く」は禁止。
