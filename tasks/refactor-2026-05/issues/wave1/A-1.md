TITLE: [security] team-hub: file_locks の normalize_path が .. と絶対 path を許可・team あたりの lock 数上限なし
LABELS: security,bug,rust

## 概要 / Summary

`team_lock_files` で受ける path の `normalize_path` は backslash 変換 / `./`-prefix 除去 / `//` 圧縮 / 末尾 `/` 削除のみで、`..` セグメントの除去や絶対 path 検出を一切しない。さらに team あたりの lock 数上限もないため、`team_lock_files({paths: ["../../etc/passwd"]})` のような traversal や、無制限の lock 登録による DoS が成立する。

## 根本原因 / Root cause

- `src-tauri/src/team_hub/file_locks.rs:73-108` の `normalize_path` で:
  - backslash → slash 変換のみ
  - `./` prefix 削除のみ
  - `//` を 1 つの `/` に圧縮
  - 末尾 `/` を除去
  - **`..` セグメントの除去や reject が無い**
  - **絶対 path (`/etc/...` / `C:\...`) の reject が無い**
- `src-tauri/src/team_hub/protocol/tools/file_lock.rs:67-82` でも team あたりの lock 数上限を設定していない (= 無制限挿入で DoS 可)。
- 結果、advisory lock として「ロック対象 path 文字列」を任意に詰められ、内部 HashMap の key 衝突による他チームの正当な lock を妨害する経路が存在する (path 文字列が一致するため、表示や別 worker からの check で誤認させられる)。

## 再現手順 / Repro

1. 任意 worker から MCP `team_lock_files({ paths: ["../../etc/passwd", "../../../C:/Windows/System32"] })` を呼ぶ。
2. lock entry が登録され、表示・解除経路で同じ文字列が流通する。
3. 別の経路で `team_lock_files({ paths: <数千件の生成 path> })` を投げ、Hub の HashMap を膨張させる。
4. team 削除 (`team_dismiss`) 経由でしか解放されないため、メモリ占有が長期化する。

## 影響範囲 / Impact

- **severity: HIGH (security / DoS / authz bypass の補助線)**
- traversal は実 file 操作には直接結びつかないが、同 path 文字列を別 IPC (`files_read` / `git_diff`) と組み合わせると、advisory lock を「許可された活動範囲」として誤認させる経路が出来る。
- lock 数無制限は Hub プロセスのメモリを単調増加させ、長期 session で OOM 起因の停止を誘発できる。
- A-7 (`terminal_create attach_if_exists` の team_id 検証) や A-8 (`handoffs_*` cross-project 検証) と組み合わせて初めて完全な path 隔離が成立する。

## 修正方針 / Fix proposal

### 案1 (推奨): normalize_path を 'rejecting' validator に変える + team あたり lock 数上限 128

1. `normalize_path` の出力を `Result<String, FileLockError>` に変更。以下を reject:
   - `..` セグメントを含む path (`Path::new(p).components().any(|c| matches!(c, Component::ParentDir))`)
   - 絶対 path (Unix の `/`-始まり / Windows の `C:\` `\\` `\\?\`)
   - 空 path / 過大長 (例: 1024 bytes 超)
   - 制御文字 (NUL / 0x01〜0x1F) を含む path
2. `src-tauri/src/team_hub/protocol/tools/file_lock.rs:67-82` 内で team あたり lock 数上限 (定数 `MAX_LOCKS_PER_TEAM = 128`) を導入。超過時は `ToolError::too_many_locks` を返す。
3. socket 異常切断時の lock 自動解放 hook (`handle_client` の return path) は別 issue (Tier C) で対応する旨を明示。
4. 既存 lock のうち `..` を含むものは migration として team_hub 起動時に best-effort で除去 (warn ログ + skip)。

```rust
// pseudocode for normalize_path
pub fn normalize_path(raw: &str) -> Result<String, FileLockError> {
    if raw.len() > 1024 { return Err(FileLockError::TooLong); }
    if raw.bytes().any(|b| b < 0x20 && b != b'\t') {
        return Err(FileLockError::ControlChar);
    }
    let p = std::path::Path::new(raw);
    if p.is_absolute() { return Err(FileLockError::Absolute); }
    if p.components().any(|c| matches!(c, std::path::Component::ParentDir)) {
        return Err(FileLockError::ParentDir);
    }
    // ... 既存の slash 正規化 ...
    Ok(normalized)
}
```

### 案2: lock 解放 TTL (60 分)

- 定期 sweep で 60 分以上更新の無い lock を解放する。
- DoS 回避のサブ手段だが上限 128 + reject の方が直接的なので、本 issue では採用せず、Tier C-14 で別途対応。
