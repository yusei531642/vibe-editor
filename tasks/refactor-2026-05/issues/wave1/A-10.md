TITLE: [security] backend: atomic_write に mode 引数を追加し mcp_config (~/.claude.json / ~/.codex/config.toml) を 0o600 強制
LABELS: security,refactor,rust,backend

## 概要 / Summary

`atomic_write` は temp file → rename で安全に書き換えるが、temp ファイル作成時のデフォルト mode が `0o644` のままで、rename 後も `~/.claude.json` / `~/.codex/config.toml` の権限が `0o644` で残る可能性がある。一方 bridge.js (`team_hub::state.rs:1041-1045`) では `0o600` を明示しており、mcp_config 系との不整合がある。`atomic_write` に optional `mode: Option<u32>` を追加して 0o600 を強制する。

## 根本原因 / Root cause

- `src-tauri/src/util/atomic_write.rs` (もしくは equivalent) の `atomic_write` 系 helper が temp file を作る際に mode を明示していない。OS デフォルト (umask 反映) で `0o644` 程度が割り当てられる。
- `tokio::fs::rename` (= `MoveFileEx` / `rename` syscall) は **temp の権限を引き継ぐ** ため、target file の旧権限 (例えば手動で `0o600` にしてあった) を上書きして緩めてしまう。
- `src-tauri/src/mcp_config/claude.rs:42` / `src-tauri/src/mcp_config/codex.rs:78` は `atomic_write` を使うが、書込み後に `set_permissions` を呼んでいない。
- 対比として `src-tauri/src/team_hub/state.rs:1041-1045` (bridge.js 配置) は `0o600` を明示している (= 内部で正しい例が既にある)。

## 再現手順 / Repro

1. `~/.claude.json` の権限を `chmod 600` で絞っておく (機密情報 = MCP server 認証 token を含む)。
2. vibe-editor 経由で team_mcp setup (= `app_setup_team_mcp`) を実行。
3. `atomic_write` 経由で `~/.claude.json` が書き換わったあと、`stat -c '%a' ~/.claude.json` で権限を確認すると `644` になっている。
4. 他のプロセスから読み取り可能になり、MCP token / 設定が露出する。

## 影響範囲 / Impact

- **severity: HIGH (security / privacy / token leak)**
- `~/.claude.json` / `~/.codex/config.toml` には MCP server 接続情報 (token / URL / scope) が含まれることがあり、user-readable なら同 user 配下の任意プロセスから読み出せる。
- A-5 (peer UID/SID 検証) と組み合わせると、user 単位の信頼境界がもう一段固くなる。
- atomic_write 自体の改修なので、影響範囲が広い (= settings / role-profiles 等の他 helper への副次効果も検証が必要)。

## 修正方針 / Fix proposal

### 案1 (推奨): atomic_write に optional mode 引数を足し、mcp_config 経路で 0o600 を渡す

1. `atomic_write(path, contents)` の signature を `atomic_write(path, contents, opts)` または `atomic_write_with_mode(path, contents, mode)` に拡張:
   ```rust
   pub struct AtomicWriteOpts {
       pub mode: Option<u32>, // Unix のみ。Windows では ACL 対策は別途
   }
   pub async fn atomic_write_with_opts(
       path: &Path,
       contents: &[u8],
       opts: AtomicWriteOpts,
   ) -> std::io::Result<()> {
       // 1. tempdir で OpenOptions::mode(opts.mode.unwrap_or(0o600)) で create
       // 2. write_all + sync_all
       // 3. tokio::fs::rename
       // 4. (defensive) tokio::fs::set_permissions(path, Permissions::from_mode(opts.mode.unwrap_or(0o600)))
   }
   ```
2. 既存 callsite は wrapper `atomic_write(path, contents)` で `AtomicWriteOpts::default()` 相当 (`mode = None` = OS デフォルト) を呼ぶよう保つ。
3. `src-tauri/src/mcp_config/claude.rs` / `src-tauri/src/mcp_config/codex.rs` の setup 経路で `atomic_write_with_opts(..., AtomicWriteOpts { mode: Some(0o600) })` を渡す。
4. Windows では `OpenOptions::mode()` 不可なので no-op でフォールバック (= プロセス所有者 only の DACL 設定が必要なら別 issue で対応)。`#[cfg(unix)]` / `#[cfg(windows)]` 分岐で安全化。
5. defensive に rename 後 `set_permissions(0o600)` も呼ぶ (= temp ファイルの mode 取りこぼしも回収)。

### 案2: mcp_config の各 setup 関数内で書込み後 `set_permissions(0o600)` を呼ぶ

- 局所修正で済むが、書込みと権限設定の間に race window が残る (rename 直後に他プロセスが読みうる)。
- 案 1 の方が race を理論上塞げるため、Wave 1 では案 1 推奨。
