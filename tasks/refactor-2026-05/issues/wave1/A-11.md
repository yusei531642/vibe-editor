TITLE: [security] updater: endpoints を二重化 + minisign 署名失敗を 1 回だけユーザー通知
LABELS: security,enhancement,rust,javascript,dependencies

## 概要 / Summary

Tauri updater の `endpoints` 配列が単一エントリのため、GitHub の TLS / DNS / Releases asset 障害時に全ユーザーが silent に更新を受け取れなくなる。また `silentCheckForUpdate` は `import.meta.env.PROD` でしか起動しないうえ、minisign 署名検証失敗を toast 1 行で隠蔽する。endpoints の二重化と、署名失敗を 24h に 1 度だけユーザー通知する経路を入れる。

## 根本原因 / Root cause

- `src-tauri/tauri.conf.json:75-80` の `plugins.updater.endpoints` が GitHub Releases の URL 1 本のみ。
- GitHub の TLS / DNS / asset 障害時、updater が成功しないが UI には「最新です」相当の表示しか出ない。
- `src/renderer/src/lib/updater-check.ts:86-87` の `silentCheckForUpdate` が `import.meta.env.PROD` でしか動かず、preview build (sign 無し) では恒常的に `signature` 系 error を握りつぶす。
- 失敗 error の種類分岐が無いため、ユーザーに「署名検証が落ちている」ことが伝わらない (= 真の updater 故障を見逃す)。

## 再現手順 / Repro

1. ホスト側で GitHub Releases (`api.github.com` または asset CDN) を hosts ファイルで block する。
2. vibe-editor を起動。
3. 自動 updater check が走るが、endpoint が 1 本のため即失敗。UI には「最新です」のみ表示。
4. minisign 署名失敗 (preview build 等) も同様に silent failure。

## 影響範囲 / Impact

- **severity: HIGH (security / availability)**
- 緊急 patch (security fix 含む) を出した直後に GitHub 障害が起きると、全ユーザーが更新を受けられない silent な状態が続く。
- minisign 署名検証失敗を隠すと、CDN 改竄や中間者攻撃を見逃すことになりかねない。
- updater 経路の信頼性は本体 vulnerability の patch 配布速度に直結するため、availability も security 視点として扱う。

## 修正方針 / Fix proposal

### 案1 (推奨): endpoints 二重化 + 24h に 1 度の署名失敗通知 + error code 別の error toast

#### A. endpoints 二重化

1. `tauri.conf.json:75-80` の `plugins.updater.endpoints` に backup 1 本を追加。例:
   ```json
   "endpoints": [
     "https://github.com/<owner>/vibe-editor/releases/latest/download/latest.json",
     "https://updates.vibe-editor.example.com/latest.json"
   ]
   ```
   - backup は CDN / S3 / 独自 host のいずれか (まずは S3 + CloudFront か、Cloudflare R2 が手軽)。
   - 両 endpoint で minisign 署名済みの同一 `latest.json` を配布。鍵は同一 (= 既存 minisign key を流用)。
   - DNS / TLS が異なる事業者間で diversify することで GitHub 障害時の継続性を担保。
2. Tauri updater は配列順に試行するため、GitHub 優先 + backup フォールバックの順で記述。

#### B. silentCheckForUpdate の error 分類 + 24h クールダウン通知

1. `src/renderer/src/lib/updater-check.ts:86-87` の `silentCheckForUpdate` で error の `kind` を見て分岐:
   - `signature` 系 error → `~/.vibe-editor/updater-last-warned.json` (ISO 8601 timestamp) を読み、24h 以上経っていれば 1 度だけ toast (`updater.signatureFailed`) を出す。
   - network 系 error → 静かに skip (頻発するため)。
   - その他 → 既存挙動。
2. cooldown ファイルは zustand persist or 直接 disk に書き出し、複数起動でも単一 source とする。
3. preview build / unsigned build の判定は `import.meta.env.PROD && import.meta.env.VITE_APP_SIGNING_ENABLED` のような flag を導入し、unsigned build では署名失敗エラーが出ない経路を確保 (toast spam 防止)。
4. i18n 文言を ja / en で追加 (`updater.signatureFailed` / `updater.endpointAllFailed`)。

### 案2: バックアップ endpoint を追加せず、署名失敗のみ通知

- minisign 失敗の通知だけ実装し、endpoints は単一のまま。
- GitHub 障害時の availability 問題が残るため、本 issue では推奨しない。
