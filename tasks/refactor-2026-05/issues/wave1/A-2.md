TITLE: [security] backend: team_state_read で active project_root 一致を検証 (cross-project leak 防止)
LABELS: security,bug,rust,backend

## 概要 / Summary

`team_state_read(project_root, team_id)` は引数の `project_root` を `project_key()` (base64) で encode してそのまま `~/.vibe-editor/team-state/<base64>/<safe_segment(team_id)>.json` を読みに行くだけで、active project_root と一致するかを検証していない。renderer 由来の任意 caller が他プロジェクトの team state を任意に閲覧可能。

## 根本原因 / Root cause

- `src-tauri/src/commands/team_state.rs:385-391` の `team_state_read` IPC 入口で、`AppState` に保存された active project_root との一致確認が行われていない。
- 同 file の `268-296` 付近の load 処理は引数 `project_root` をそのまま base64 encode して disk path を構築する。
- `app_setup_team_mcp` 等の他 IPC では state 経由 (`state.project_root.lock().await`) で active project_root を取得する流儀があるが、`team_state_read` だけ素通り。
- 他にも `team_diagnostics_read` (A-3) / `handoffs_*` (A-8) で同種の検証漏れがあり、本 issue の修正パターンを共通化することで一括対応できる。

## 再現手順 / Repro

1. vibe-editor を `~/projectA` で起動し、team を recruit して team-state を生成する。
2. 別の起動で `~/projectB` に切り替える。
3. renderer から細工した `invoke('team_state_read', { projectRoot: '/Users/me/projectA', teamId: 'team-xxxxxx' })` を呼ぶ。
4. `~/.vibe-editor/team-state/<base64-of-projectA>/team-xxxxxx.json` の内容が読み出され、Cross-project の team 状態 (member 一覧 / role / agent_id / 課題進捗) が漏洩する。

## 影響範囲 / Impact

- **severity: HIGH (data leak / authz bypass)**
- 他プロジェクトの team 構成 (agent_id / role / handoff 履歴の summary) が任意 caller に流出する。
- malicious extension / compromised renderer / dev-tools 経由の調査において、disk 上の base64 encoded project root を逆算しなくても renderer から直接読める点が重大。
- `team_diagnostics_read` (A-3) と組み合わせると、active project と無関係な team の `serverLogPath` まで取得可能。

## 修正方針 / Fix proposal

### 案1 (推奨): active project_root 一致を共通 helper で検証

1. `src-tauri/src/commands/authz.rs` 等を新設 (or 既存 `commands/mod.rs` 内) し、以下のような helper を追加:
   ```rust
   pub async fn assert_active_project_root(state: &AppState, given: &str) -> Result<PathBuf, CommandError> {
       let active = state.project_root.lock().await.clone()
           .ok_or(CommandError::NoActiveProject)?;
       let active_canon = tokio::fs::canonicalize(&active).await
           .map_err(|_| CommandError::Authz("project_root canonicalize failed"))?;
       let given_canon = tokio::fs::canonicalize(given).await
           .map_err(|_| CommandError::Authz("given project_root canonicalize failed"))?;
       if active_canon != given_canon {
           return Err(CommandError::Authz("project_root mismatch"));
       }
       Ok(active_canon)
   }
   ```
2. `team_state_read` の冒頭で `assert_active_project_root(&state, &args.project_root).await?` を呼ぶ。
3. 同種の保護を `team_diagnostics_read` (A-3) / `handoffs_*` (A-8) でも同 helper で実装。

### 案2: 引数 `project_root` を IPC から削り、active project_root を Rust 側で固定的に使う

- API 互換性が壊れるが、誤用余地が完全になくなる。
- マルチプロジェクト並列で 1 vibe-editor インスタンスを使うユースケースを想定するか確認してから採否判断。
- 本 issue では非推奨 (案 1 の方が破壊的変更なし)。
