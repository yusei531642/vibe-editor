TITLE: [security] backend: team_diagnostics_read を active team set に限定 (renderer impersonation 防止)
LABELS: security,bug,rust,backend

## 概要 / Summary

`team_diagnostics_read` IPC は renderer の任意 caller に対して、Hub 内部で `role: "leader"` の `CallContext` を組み立てて `team_diagnostics()` を通すため、ViewDiagnostics permission check が常時 bypass される。任意 team_id の `serverLogPath`、全 member の `agent_id`、`recruitedAt` 等が漏洩する。

## 根本原因 / Root cause

- `src-tauri/src/commands/team_diagnostics.rs:24-41` で renderer 由来の `teamId` をそのまま受け、Hub 内部で `CallContext { role: "leader", ... }` を捏造して `team_diagnostics()` に通している。
- 結果として、Hub 側で実装されている permission check (`ViewDiagnostics` 相当) は role が leader 固定のため常に true となる。
- active な team set (`state.team_hub.state.active_teams` 相当) との照合が行われていないため、過去 / 別プロジェクトの team_id まで参照可能。
- renderer の信頼境界の外から「leader として振る舞う」ことができる構造になっている。

## 再現手順 / Repro

1. vibe-editor で team A を 1 度作って終了する (disk に team-state が残る)。
2. 別 project / 別タイミングで vibe-editor を起動する。
3. renderer DevTools から `invoke('team_diagnostics_read', { teamId: 'team-of-projectA' })` を呼ぶ。
4. `serverLogPath` (= 環境変数 `VIBE_TEAM_LOG_PATH`) や全 member の `agent_id`, `recruitedAt`, `currentStatus`, role bind 履歴が JSON で返る。

## 影響範囲 / Impact

- **severity: HIGH (data leak / impersonation / log path disclosure)**
- 他プロジェクト / 過去の team の構成と健康状態を任意に閲覧できる。
- `serverLogPath` 漏洩は `VIBE_TEAM_LOG_PATH` の絶対 path と user 名を露出する (privacy + recon)。
- malicious extension の場合、`teamId` を probe して存在/非存在を判定する recon ができる。
- A-2 (`team_state_read` 経路) と組み合わせると、cross-project の team 構成が完全に観測可能になる。

## 修正方針 / Fix proposal

### 案1 (推奨): active team set に含まれる team_id だけ受理 + 不一致は 404 相当を返す

1. `team_diagnostics_read` 冒頭で:
   ```rust
   let hub_state = state.team_hub.state.lock().await;
   if !hub_state.active_teams.contains_key(&args.team_id) {
       return Err(CommandError::Authz("team_id is not active"));
   }
   ```
2. さらに active project_root との一致確認 (A-2 の helper を再利用) を入れて、別 project 由来の team_id を除外。
3. Hub 側 `CallContext` を「leader 固定」から「renderer trusted caller (read-only diagnostics)」に変更し、ViewDiagnostics 相当の permission check で role 付与する。permission flag を `diagnostics_renderer: true` のような独立 capability で扱うのが望ましい。
4. error 応答は team_id の存在/非存在を区別しない (`Authz("denied")` 統一) ことで recon を抑制。

### 案2: renderer caller を Tauri permission cap で限定

- Tauri 2.x の per-command permission scope を使い、`team_diagnostics_read` を main window のみ呼べる cap に絞る。
- renderer 内部の compromise には依然弱いが、外部 plugin / iframe からの呼び出しを完全に遮断できる。
- 案 1 と直交、両方適用するのが望ましい (in-depth defense)。
