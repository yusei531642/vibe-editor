TITLE: [security] team-hub: socket / named pipe で peer UID/SID 検証を追加 (token 盗難の局所封じ込め)
LABELS: security,bug,rust

## 概要 / Summary

team-hub の Unix socket / Windows named pipe handshake は token (24 byte hex 48 文字) のみを検証し、peer の UID (Unix `SO_PEERCRED`) や SID (Windows `GetNamedPipeClientProcessId`) を検査していない。token は環境変数 `VIBE_TEAM_TOKEN` 経由で child process に渡されるため、同 user の任意のローカルプロセスが token を盗み見れば handshake を成立させられる。

## 根本原因 / Root cause

- `src-tauri/src/team_hub/mod.rs:78-94, 144-216` の handshake パスで、token 比較しか行っていない。
- token は env 経由で worker に渡るため、`/proc/<pid>/environ` (Linux) や `Get-Process | Select-Object -ExpandProperty StartInfo` (Windows) 等で同一 user の他プロセスから観測しうる。
- Unix socket での `getsockopt(SOL_SOCKET, SO_PEERCRED, ...)` (Linux) / `getpeereid` (BSD/macOS) や、Windows named pipe での `GetNamedPipeClientProcessId` → `OpenProcessToken` → `GetTokenInformation(TokenUser)` による client SID 取得が一切行われていない。
- 結果、token 盗難で「同 user 内の任意のローカルプロセス」が Hub への接続を確立できる。

## 再現手順 / Repro

1. vibe-editor を起動して team を作る (Hub が VIBE_TEAM_TOKEN を発行)。
2. 同 user の別プロセス (例: 悪性 IDE plugin / 別 terminal の cat /proc/<vibe-editor pid>/environ) から token を読み取る。
3. 攻撃者プロセスから Unix socket (`~/.vibe-editor/team-hub.sock`) または Windows pipe (`\\.\pipe\vibe-editor-team-hub-<...>`) に接続し、handshake `hello` line を送る。
4. token が一致するため接続が成立し、攻撃者プロセスが任意 MCP tool を呼び出せる。

## 影響範囲 / Impact

- **severity: HIGH (authz / token theft の被害局所化失敗)**
- 同 user の 1 プロセスが侵害されると、token 経由で Hub 接続まで連鎖する (= 横展開が容易)。
- vibe-editor の team-hub には `team_assign_task` / `team_send` / `team_dismiss` 等 worker 操作の中核が露出しているため、token 盗難 1 件で team 全体の制御が奪われる。
- root 権限まで上げる経路ではないが、「user 権限内であればなんでもできる」になりやすい。

## 修正方針 / Fix proposal

### 案1 (推奨): Unix で getpeereid 検証、Windows で GetNamedPipeClientProcessId + OpenProcessToken 検証

#### Unix
1. accept 直後に `getsockopt(SOL_SOCKET, SO_PEERCRED, ...)` (Linux) / `getpeereid` (BSD/macOS) で peer UID を取得。
2. `nix::unistd::getuid()` で現在の vibe-editor プロセス自身の UID を取得し、不一致なら即切断 + warn ログ。
3. 失敗 (= getpeereid syscall に失敗) も同様に reject。

```rust
// pseudocode (Linux)
#[cfg(target_os = "linux")]
fn check_peer_uid(stream: &UnixStream) -> Result<(), HandshakeError> {
    use nix::sys::socket::{getsockopt, sockopt::PeerCredentials};
    let cred = getsockopt(stream.as_raw_fd(), PeerCredentials)
        .map_err(|_| HandshakeError::PeerCheckFailed)?;
    if cred.uid() != nix::unistd::getuid().as_raw() {
        return Err(HandshakeError::PeerUidMismatch);
    }
    Ok(())
}
```

#### Windows
1. `GetNamedPipeClientProcessId(handle, &mut pid)` で接続元 PID を取得。
2. `OpenProcessToken(OpenProcess(QUERY_LIMITED_INFORMATION, false, pid), TOKEN_QUERY, &mut token)` で token を取得。
3. `GetTokenInformation(token, TokenUser, ...)` で client SID を取得。
4. `GetCurrentProcessToken` から取った own SID と比較し、不一致なら切断。
5. PID 取得から OpenProcessToken まで race 余地があるため、PID は `IsProcessActive` のような確認段階を併設するか、SID 比較を最終段に置く。

#### 共通
- handshake 失敗を rate-limit して「token brute force + 反復試行」が現実的でなくなるよう抑制 (例: 同 socket からの失敗 5 回で 60 秒切断)。
- token は random 24 byte hex で十分長いが、env 経由で漏れることを前提に「peer 認証」を上に積む。

### 案2: token を env 経由 ではなく ANCILLARY data で受け渡し

- spawn 時に socket pair を継承して fd 経由で token を渡す。
- Windows では DuplicateHandle 経由で同等のことをする。
- env からの漏洩は完全に塞げるが、子プロセス側 (Claude / Codex CLI) の協力が必要で侵襲が大きい。
- 案 1 と組み合わせて漸進的に進める長期改善。Wave 1 では案 1 のみで十分。
