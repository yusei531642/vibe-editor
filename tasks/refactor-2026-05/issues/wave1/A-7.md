TITLE: [security] backend: terminal_create attach_if_exists で team_id 一致を検証 (scrollback 漏洩防止)
LABELS: security,bug,rust,backend

## 概要 / Summary

`terminal_create` の `attach_if_exists` 経路は renderer 由来の任意 `session_key` / `agent_id` で他 team / 他 session に attach できる。PTY scrollback には Claude Code の prompt / API キー / git diff / ファイル内容が混入しうるため、別 team の scrollback を吸い出す情報漏洩経路となる。

## 根本原因 / Root cause

- `src-tauri/src/commands/terminal.rs:169-207` の `find_attach_target` / attach 経路で、attach 候補に対して呼び出し側が指定した `team_id` (現状 `opts.team_id`) と完全一致するかを検証していない。
- `session_key` / `agent_id` の文字列一致だけで attach するため、異なる team の同名 agent_id があれば誤接続する。
- PTY scrollback は ConPTY の出力を全保持しており、Claude Code の prompt 履歴や git diff、コードベースの ファイル内容が文字列として残っている (= 機微情報の塊)。

## 再現手順 / Repro

1. team A で worker `programmer-1` を起動して、機微なコードや git diff を表示する。
2. 別 team B で同じ `agent_id` 文字列 (= `programmer-1` の正規化済み id) を使って renderer 側から `invoke('terminal_create', { sessionKey: 'agent:programmer-1', agentId: 'programmer-1', teamId: 'team-B', attachIfExists: true })` を呼ぶ。
3. team A の既存 PTY に attach され、scrollback として team A の機微情報が renderer に流れる。

## 影響範囲 / Impact

- **severity: HIGH (data leak / authz)**
- 別 team (= 別ユーザーフロー / 別プロジェクト) の scrollback が吸い出される。
- scrollback には API キー / 認証情報 / git diff / ファイル内容など、最も機微なテキストが含まれることが多い。
- attach は read-only ではなく write も可能 (write 経路はさらに別問題、`terminal_write` の id validate 欠落と組み合わさるとさらに悪化)。

## 修正方針 / Fix proposal

### 案1 (推奨): attach 候補エントリに team_id を保持し、不一致なら新規 spawn にフォールバック

1. `pty::registry` の attach target エントリに `team_id: Option<String>` フィールドを追加。spawn 時に紐付けて保存。
2. `find_attach_target` の戻り値で `team_id` を返し、attach リクエストの `opts.team_id` と完全一致しない場合は `None` を返して attach しない。
3. attach できなかった場合は通常の新規 spawn 経路にフォールバック (UX を壊さない)。
4. team_id が `None` (= 単独 PTY) の場合は team_id 不一致でも attach 不可とし、将来必ず明示的に team_id を持たせる migration を進める。
5. attach reject 時に `tracing::info!` 程度で agent_id / 試行 team_id を audit log に残す (warn は新規 spawn fallback が常態化すると noisy なので info)。

```rust
// pseudocode
if let Some(target) = find_attach_target(&session_key, &agent_id, &opts.team_id) {
    if target.team_id.as_deref() != Some(&opts.team_id) {
        // mismatch -> fallback to new spawn
        spawn_new_pty(...).await?;
    } else {
        attach(target).await?;
    }
}
```

### 案2: session_key 自体に team_id を encode する

- session_key を `team-<team_id>:agent:<agent_id>` 形式に変えて、key 文字列で衝突しないように構造化する。
- 既存 key の互換性を壊すため migration が必要。
- 案 1 の方が侵襲的でなく、Wave 1 では非推奨 (将来 refactor で検討)。
