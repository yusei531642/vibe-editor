TITLE: [security] backend: handoffs_* IPC で active project_root 一致を検証 (cross-project read 防止)
LABELS: security,bug,rust,backend

## 概要 / Summary

`handoffs_list` / 関連する `handoffs_*` IPC は project_root / team_id を `safe_segment` / `project_key` で encode するだけで、handoff ディレクトリ越境 (= active project_root と不一致な project_root を素通しで読む) のチェックが無い。renderer 由来の任意 caller が他プロジェクトの handoff 履歴を読み出せる。

## 根本原因 / Root cause

- `src-tauri/src/commands/handoffs.rs:115-138, 338-362, 365-374` (`handoffs_list` 周辺) で、引数 `project_root` を `project_key()` (base64) で encode し `~/.vibe-editor/handoffs/<base64>/<safe_segment(team_id)>/...` を直接読む。
- `state.project_root.lock().await` 経由の active project_root との一致確認が一切行われていない。
- A-2 (`team_state_read`) / A-3 (`team_diagnostics_read`) と同型のクロスプロジェクト leak。

## 再現手順 / Repro

1. vibe-editor を `~/projectA` で起動し、worker から複数 handoff を発行する (= disk に履歴が残る)。
2. 別タイミングで `~/projectB` を開く。
3. renderer DevTools から `invoke('handoffs_list', { projectRoot: '/Users/me/projectA', teamId: 'team-xxx' })` を呼ぶ。
4. project A の handoff 履歴 (送信者 / 受信者 / 本文 summary / timestamp) が JSON で返る。

## 影響範囲 / Impact

- **severity: HIGH (data leak / authz bypass)**
- handoff 本文には team_send で渡された data も含まれる (= プロンプト本文 / 機微テキスト)。
- 過去の検索 / 復旧目的で handoff を残しているプロジェクトでは、過去の機密が筒抜けになる。
- A-2 / A-3 と同じ helper で集中的に保護できるため、本 issue を含めた一括対応が望ましい。

## 修正方針 / Fix proposal

### 案1 (推奨): A-2 で導入する `assert_active_project_root` helper を再利用

1. A-2 で導入する `commands/authz.rs::assert_active_project_root(state, given) -> Result<PathBuf>` を `handoffs_*` 系 IPC 入口に挿入する。
2. 対象 IPC:
   - `handoffs_list`
   - `handoffs_read` (もし存在すれば)
   - `handoffs_save` / `handoffs_append` 等の write 系も同様
   - `recruit_observed_while_hidden` (project_root 引数を取る系)
3. 不一致なら `CommandError::Authz` で reject し、空 list / `None` を返す経路ではなく明示的な error にする (silent fail は呼び出し側の bug 検出を遅らせる)。
4. error 応答は team_id / project_root の存在/非存在を区別しない統一文言 (= recon 抑止)。

```rust
// pseudocode
#[tauri::command]
pub async fn handoffs_list(
    state: tauri::State<'_, AppState>,
    args: HandoffsListArgs,
) -> Result<Vec<HandoffEntry>, CommandError> {
    crate::commands::authz::assert_active_project_root(&state, &args.project_root).await?;
    // 既存処理を続行
}
```

### 案2: project_root 引数を IPC から削り、Rust 側の active で固定

- A-2 と同じく、API 互換性を壊すリスクがある。
- マルチプロジェクト並列での運用が想定されるなら案 1 で十分。
- Wave 1 では非推奨。
