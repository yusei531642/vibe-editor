TITLE: [security] terminal: --resume <id> 注入時の sessionId を UUID-form で validate
LABELS: security,bug,rust

## 概要 / Summary

PTY 起動時の `--resume <id>` 引数に渡される `resumeSessionId` は、renderer (zustand persist の `team-history.json` / `~/.claude/projects/<encoded>/<id>.jsonl` の file_stem) 由来で、validate なしに argv に挿入される。`String` を 1 trust step で argv に詰めるため、`-` 始まりや shell metachar / 改行を含む id を埋めると引数注入や parse 破壊が成立する。

## 根本原因 / Root cause

- `src/renderer/src/components/canvas/cards/AgentNodeCard/CardFrame.tsx:336-340` 周辺で renderer 側から `resumeSessionId` を `terminal_create` の args に詰めている。
- Rust 側 `terminal_create` で `--resume <id>` を構築する直前に、id が UUID 形式 / 安全 charset であることを確認していない。
- `~/.claude/projects/<encoded>/*.jsonl` の file_stem は通常 UUID v4 だが、悪意ある書込みで `--evil-flag` や `; rm -rf ~` のような id を作ることが理論上可能。
- zustand persist の `team-history.json` も renderer-writable なので、ブラウザ DevTools 経由で直接書き換えられる。

## 再現手順 / Repro

1. `~/.vibe-editor/team-history.json` または `~/.claude/projects/<encoded>/` を手動編集して、resumeSessionId として `--print=/etc/passwd` のような文字列を埋め込む。
2. Canvas で対応する worker を resume する。
3. argv に `--resume --print=/etc/passwd` のような連結が入り、Claude / Codex CLI 側の引数 parse が壊れる (= 任意のフラグ注入)。
4. file_stem に改行を入れた場合、ConPTY 側のコマンドラインが分割されて別コマンドの起動に化ける可能性 (Windows の処理系依存)。

## 影響範囲 / Impact

- **severity: HIGH (security / argument injection)**
- 主に同 user 内の renderer compromise から起きる経路だが、攻撃成立後の impact が大きい (任意フラグで Claude/Codex の挙動を変えられる)。
- file_stem 由来のため、`~/.claude/projects/` をクラウド sync しているユーザーで sync 元から汚染される可能性も。
- A-7 (attach team_id 検証) と組み合わせて初めて完全な argv 安全性が確保される。

## 修正方針 / Fix proposal

### 案1 (推奨): Rust 側 `terminal_create` 入口で正規表現 validate + reject

1. `src-tauri/src/commands/terminal.rs` の `terminal_create` で resumeSessionId を扱う直前に以下を挿入:
   ```rust
   // pseudocode
   const RESUME_ID_RE: &str = r"^[A-Za-z0-9_-]{8,64}$";
   fn validate_resume_session_id(id: &str) -> Result<(), CommandError> {
       static RE: OnceLock<Regex> = OnceLock::new();
       let re = RE.get_or_init(|| Regex::new(RESUME_ID_RE).unwrap());
       if !re.is_match(id) {
           return Err(CommandError::Validation("invalid resumeSessionId format"));
       }
       Ok(())
   }
   ```
2. validate NG なら resume を諦めて新規起動にフォールバックするか、明示的に error を返すかは UX 判断。**推奨**は新規起動フォールバック (UX 維持) + `tracing::warn!` で id を切り詰めて audit log に残す。
3. UUID v4 限定 (8-4-4-4-12 hex) にしても良いが、`~/.claude/projects/` の id 形式が将来変わる可能性を考慮し `[A-Za-z0-9_-]{8,64}` の緩い whitelist 推奨。
4. argv 連結時に `--resume=<id>` ではなく **完全に独立した argv 要素** として渡すことを保証 (`Command::arg("--resume").arg(&id)` のように 2 要素)。

### 案2: renderer 側でも事前 validate

- 二段階 defense として renderer 側でも同 regex で validate し、disk 由来の汚染を入口で弾く。
- Rust 側の validate が真の防御なので、renderer 側は UX 改善 (即時フィードバック) として位置付ける。
- 案 1 と組み合わせる前提で実装。
