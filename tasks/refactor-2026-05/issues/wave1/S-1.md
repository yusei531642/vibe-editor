TITLE: [security] team-hub: team_update_task に assignee 検証を追加して任意 worker による task done 化を防ぐ
LABELS: security,bug,rust

## 概要 / Summary

`team_update_task` が caller の権限・assignee 一致を検証しないため、同 team に属する任意の worker が他者 task を `done` / `blocked` / `human_gate` に変更し、`done_evidence` まで捏造可能。`team_report` (#572) で同等の穴を塞いだ際に、`update_task` 経路だけ修正が漏れた状態となっている。

## 根本原因 / Root cause

- `src-tauri/src/team_hub/protocol/tools/update_task.rs:156-292` の `team_update_task` 関数で、`check_permission` も assignee 一致検証も呼ばれていない。
- 受信した `task_id` で team / task を引き当てたあと、215 行付近で即 `task.status = status.to_string()` を実行。caller (`ctx.role` / `ctx.agent_id`) と `task.assigned_to` の関係を一切問わない。
- 一方 `src-tauri/src/team_hub/protocol/tools/report.rs:285` の `team_report` では既に「`task.assigned_to == ctx.role || task.assigned_to == ctx.agent_id || ctx.role == "leader"` でなければ task 改竄を拒否」が実装済みで、本来同等のガードが update_task にも必要。

## 再現手順 / Repro

1. 同一 team に worker A (role=`programmer-1`) と worker B (role=`reviewer-1`) を recruit する。
2. Leader が `team_assign_task({ assigned_to: "reviewer-1", ... })` で task X を発行する。
3. worker A が直接 MCP `team_update_task({ task_id: X, status: "done", done_evidence: [{criterion: "fabricated", evidence: "..."}] })` を呼ぶ。
4. 検証なしで status が `done` に書き換わり、Leader 側のチェックリストが満たされた状態に偽装される。

## 影響範囲 / Impact

- **severity: CRITICAL (security / authz bypass)**
- 同 team の任意 worker が他者の task を任意の status へ書き換えられる (権限昇格)。
- `done_evidence` の捏造で reviewer の承認サイクルを完全 bypass 可能 → orchestration の信頼性が崩壊。
- Leader が "chain-of-responsibility" を信用しているため、誤った決断 (例: deploy 判断) を引き起こす。
- `team_report` (#572) で導入した assignee 検証の補助線を完成させないと、対称的でない権限モデルが残り続ける。

## 修正方針 / Fix proposal

### 案1 (推奨): `team_report` と同形のガードを update_task 冒頭に追加

`src-tauri/src/team_hub/protocol/tools/update_task.rs` の task 引き当て直後 (status を書く前) に以下を挿入する。

```rust
// pseudocode
let is_leader = ctx.role.as_deref() == Some("leader");
let is_assignee = task.assigned_to.as_deref() == Some(&ctx.role)
    || task.assigned_to.as_deref() == Some(&ctx.agent_id);
if !is_leader && !is_assignee {
    return Err(ToolError::permission_denied(
        "team_update_task: task is not assigned to this caller",
    ));
}
```

検証ロジックは `team_hub::protocol::authz` のような共通 helper (`fn assert_task_owner_or_leader(task, ctx) -> Result<()>`) として切り出し、`update_task` / `report` の双方から呼ぶ。重複コードを残さない。

### 案2: `check_permission(Permission::UpdateOwnTask)` を経由

`Permission` enum に `UpdateOwnTask` を追加し、role-profiles 側でロール毎に許可を宣言。柔軟性は高いが既存 perm 体系の整合チェックが広範に及ぶため Wave 1 では推奨しない。
