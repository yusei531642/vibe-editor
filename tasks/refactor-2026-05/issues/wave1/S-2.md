TITLE: [bug] canvas: EditorCard の未保存内容が close / Clear で確認なく失われる
LABELS: bug,canvas,javascript

## 概要 / Summary

Canvas モードの EditorCard は `content` / `original` をローカル `useState` で保持しており、Canvas store にも IDE 側の dirty registry にも乗っていない。カードの × ボタン、`Clear` 操作、preset の上書きで dirty な編集内容が確認なく失われる data-loss バグ。IDE 側の `confirmDiscardEditorTabs` 相当が Canvas 経路で完全に欠落している。

## 根本原因 / Root cause

- `src/renderer/src/components/canvas/cards/EditorCard.tsx:18-93` で `content` / `original` を **コンポーネントローカル `useState`** で保持しており、unmount 時に破棄される。
- `src/renderer/src/components/canvas/CardFrame.tsx:60-67` の close ボタン → `useConfirmRemoveCard(id)` → `useCanvasStore.removeCard` (`stores/canvas.ts:199-227`) に dirty 検査が一切無い。
- `src/renderer/src/layouts/CanvasLayout.tsx:509-516` の `Clear` 処理は 1 度の confirm 後に全ノードを `clear()` するため、複数 EditorCard の未保存内容が一括で消失する。
- IDE 側で同等の保護を担っている `confirmDiscardEditorTabs` (file tab 群が dirty なら一括確認するロジック) は Canvas 経路を通らない。

## 再現手順 / Repro

1. Canvas モードに切り替えて、適当なファイルを EditorCard として開く。
2. テキストを編集する (`isDirty` 相当の状態を作る)。
3. 以下のいずれかを実行する:
   - カード右上の × を押す (single confirm が出るが dirty 検査なし)
   - HUD / 右クリックメニューから `Clear` を実行する (Canvas 全体を 1 confirm でクリア)
   - 既存の team preset を別 preset で上書きする
4. 編集内容が確認なく破棄される。

## 影響範囲 / Impact

- **severity: CRITICAL (data-loss)**
- ユーザー操作 1 アクションで未保存編集を失う典型的な data-loss UX。
- 複数 Editor を並べる Canvas のユースケースほど被害が増幅する。
- IDE 側との一貫性を欠き、「Canvas は永続性が弱い」という誤った印象を強化する。

## 修正方針 / Fix proposal

### 案1 (推奨): Canvas 内 EditorCard を IDE と同じ dirty registry に統合

1. `useFileTabs` / IDE 側 dirty 管理に EditorCard を participate させる。EditorCard 側で `useFileTabs().registerEditor({ id, getDirtySnapshot })` のような hook を提供し、`isDirty` / `getCurrentContent` を中央 registry に exposed する。
2. `useConfirmRemoveCard(id)` で対象が editor かつ dirty なら、追加の confirm ダイアログを表示する (i18n: `editor.confirmDiscardChanges`)。
3. `Clear` (CanvasLayout.tsx:509-516) では「dirty editor の一覧」を取得し、件数があれば confirm dialog でリスト提示。

### 案2 (中期): EditorCard の content / original を Canvas store に持たせる

- payload に `editorState: { content, original, isDirty }` を入れて `setCardPayload` 経由で更新。
- close 経路で `payload.editorState.isDirty` を見て confirm を出す。
- 副作用: zustand persist で dirty content も autosave されるので、ブラウザ再起動でも復元できる。
- 大きいファイルで persist 容量が膨らむため、`isDirty=false` のときは `content` を payload から落とす。

### 案3 (長期): autosave / debounce write

- 1.5s debounce で `files_write` を呼び、`expected_content_hash` を更新する (#119 と整合)。
- 案 1+2 と組み合わせると最も堅牢だが、Wave 1 では案 1 のみで十分に data-loss を塞ぐ。
