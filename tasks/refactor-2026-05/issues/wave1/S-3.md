TITLE: [bug] canvas: AgentNodeCard の unreadInboxCount が連続 handoff で undercount する (closure stale)
LABELS: bug,canvas,javascript

## 概要 / Summary

`useTeamHandoff` / `useTeamInboxRead` の inbox count 更新が closure-captured `payload` を読んでから書き戻すため、1 frame (16ms) 以内に同 agentId へ複数 handoff が来ると stale な `unreadInboxCount` を base に書いてしまい、最終値が undercount する。督促判定 (worker への nudge / dead detection) の前提が壊れる。

## 根本原因 / Root cause

- `src/renderer/src/components/canvas/cards/AgentNodeCard/CardFrame.tsx:559-579` (`useTeamHandoff` 周辺) で:
  ```ts
  setCardPayload(id, { unreadInboxCount: (payload.unreadInboxCount ?? 0) + 1 });
  ```
  と書いており、`payload` は useCallback の closure で captured されている。closure は React commit 後にしか更新されない。
- `src/renderer/src/components/canvas/cards/AgentNodeCard/CardFrame.tsx:580-606` (`useTeamInboxRead`) でも同パターン。
- 同じ agentId へ短時間 (<16ms) に 2 件以上の handoff inject が来ると、両 callback がともに `payload.unreadInboxCount = 0` を見て `0 + 1 = 1` を書き、最終値が 1 件分しか反映されない。
- Leader が `team_send` を一斉に複数 worker へ broadcast したケースでも顕在化しやすい (TeamHub 側で 1ms 以内に立て続けの emit がある)。

## 再現手順 / Repro

1. Canvas で leader + 複数 worker を配置する。
2. Leader から worker A へ `team_send({ to_role: "programmer-1", ... })` を 2 件、十数 ms 以内に連続で送る (Hub 側 broadcast や handoff replay でも同じ条件が再現する)。
3. AgentNodeCard `programmer-1` の unread badge を確認すると `2` であるべきところ `1` で止まる。
4. これに `useTeamInboxRead` (受信 acknowledged) が同様の race で 1 巻き戻し → 最悪 0 を表示する。

## 影響範囲 / Impact

- **severity: CRITICAL (correctness / silent undercount)**
- 督促 / dead detection の根拠 (`unreadInboxCount` が一定時間 0 のまま) が破壊され、誤った "alive" 判定で stuck を見逃す。
- broadcast / replay 経路で常時 race するため、運用中の頻度が高い。
- bug が UI 上の数字でしか見えないため気付きにくい (silent failure)。

## 修正方針 / Fix proposal

### 案1 (推奨): functional update で zustand store の最新値を読む

closure を通さず、setter 内で直接 store を読む形に変える。

```ts
// pseudocode
const handleHandoff = useCallback(() => {
  const current = useCanvasStore
    .getState()
    .nodes
    .find((n) => n.id === id)?.data?.payload?.unreadInboxCount ?? 0;
  setCardPayload(id, { unreadInboxCount: current + 1 });
}, [id]);
```

`useTeamInboxRead` (CardFrame.tsx:580-606) でも同じ pattern を適用する。`setCardPayload` 自体に `(prev) => next` の overload を追加して中央化する案 (案2) と組み合わせると、他 callsite の予防にもなる。

### 案2: `setCardPayload` に functional update overload を導入

```ts
// stores/canvas.ts
setCardPayload(id, (prev) => ({ ...prev, unreadInboxCount: (prev.unreadInboxCount ?? 0) + 1 }));
```

- 全 setCardPayload callsite を grep して closure で payload を読んでから書く形を順次置換する。
- 一気に置換せず、まず本 issue では unreadInboxCount 周辺だけを functional 化し、別 refactor issue で全体置換する手も可。
