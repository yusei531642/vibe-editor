TITLE: [bug] backend: codex 側の MCP setup 失敗時に rollback できない (claude only snapshot で半端書き残存)
LABELS: bug,rust,backend

## 概要 / Summary

`app_setup_team_mcp` は claude 設定のみ pre-snapshot を取得し、claude → codex の順に書き込む。codex 側で失敗しても `claude::restore` だけが走り codex 側は中途半端な書込みが残る。codex には `snapshot/restore` API そのものが存在しないため、`~/.codex/config.toml` の他セクションを破壊する潜在リスクもある。`cleanup_team_mcp` 経路も同じ片肺 rollback。

## 根本原因 / Root cause

- `src-tauri/src/commands/app/team_mcp.rs:131-170` の `setup` で取られている snapshot は `claude::snapshot()` のみ。
- `src-tauri/src/mcp_config/codex.rs:60-92` には `setup` / `cleanup` 相当しか実装されておらず、`snapshot()` / `restore(snap)` が存在しない (`grep -n "snapshot" src-tauri/src/mcp_config/codex.rs` でヒット 0)。
- 失敗時 fallback は claude のみを巻き戻す形になっているため、claude → codex の途中で codex 側の write が部分成功した場合に半端な mcp 設定が残留する。
- `cleanup_team_mcp` も同様で、codex 側だけ削除に失敗すると claude 側だけが綺麗な状態に戻り、不整合が永続化する。

## 再現手順 / Repro

1. `~/.codex/config.toml` に予め大きな mcp section + 関連 server entry を配置し、書き込み権限を取り上げる (`chmod -w`) など、`codex::setup` が確実に失敗する状態にする。
2. Canvas / IDE から `app_setup_team_mcp` を呼ぶ (e.g. team preset を Apply)。
3. claude 側の write は成功するが、codex 側で書込みが失敗。
4. `~/.claude.json` だけ team-mcp が反映され、`~/.codex/config.toml` は半端 (またはエラー前の状態) のまま残る。

逆順 (codex 先成功 → claude 失敗) でも codex 側だけ rollback されないので、いずれの順でも片側残留は起こりうる。

## 影響範囲 / Impact

- **severity: CRITICAL (data integrity / config corruption)**
- ユーザーの `~/.codex/config.toml` に予期せぬ mcp server entry が **永続的に残存** する可能性がある。
- claude / codex 間で「どちらの設定が真実か」が一致しなくなる → recruit が失敗する / Codex CLI が起動できなくなる事故。
- `cleanup_team_mcp` でも片肺 rollback のため、ユーザーが手動で `~/.codex/config.toml` を編集して回復しない限り直らない。
- silent failure になりやすく、bug 報告までの lag が長い。

## 修正方針 / Fix proposal

### 案1 (推奨): codex に snapshot / restore API を新設し、claude と対称な 2-phase 書き込みにする

1. `src-tauri/src/mcp_config/codex.rs` に以下を追加:
   ```rust
   pub fn snapshot() -> Result<CodexSnapshot> { /* config.toml を tokio::fs::read で取得し、Vec<u8> として保持 */ }
   pub fn restore(snap: CodexSnapshot) -> Result<()> { /* atomic_write で原子的に書き戻す */ }
   ```
   `CodexSnapshot` は `Option<Vec<u8>>` (= 元々ファイルが存在しなかった場合は None) を保持。
2. `src-tauri/src/commands/app/team_mcp.rs:131-170` を以下のシーケンスに変える:
   ```rust
   let claude_snap = claude::snapshot()?;
   let codex_snap = codex::snapshot()?;
   match async {
       claude::setup(&team_id, ...).await?;
       codex::setup(&team_id, ...).await?;
       Ok::<_, Error>(())
   }.await {
       Ok(_) => Ok(()),
       Err(e) => {
           let _ = claude::restore(claude_snap);
           let _ = codex::restore(codex_snap);
           Err(e)
       }
   }
   ```
3. `cleanup_team_mcp` も同じ rollback 対称シーケンスに変更。

副利点:
- 2 ファイルとも atomic_write を経由するため、A-10 (`atomic_write` mode 引数) が入れば 0o600 強制も同じ経路に乗る。
- 将来 engine が増えた場合 (`gemini` など) も同 pattern で拡張できる。

### 案2: codex 側 setup を「冪等差分 patch」に変えて rollback を不要にする

- 既存 toml を read → in-memory patch → atomic_write の 1 段階にし、失敗時は何も書かないようにする。
- snapshot を持たないが「途中で失敗 = ファイルは元のまま」を保証する。
- 実装面で codex の `config.toml` 全体を完全 round-trip parse できる必要があり、TOML preserve コメントの問題 (`toml_edit` クレート導入) で工数が膨らむ。本 issue では推奨しない。
