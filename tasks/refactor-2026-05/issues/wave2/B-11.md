TITLE: [bug] pty: inject_codex_prompt_to_pty が tokio worker をブロックする (spawn_blocking 経由に揃える)
LABELS: bug,rust,backend

## 概要 / Summary

`commands/terminal.rs::inject_codex_prompt_to_pty` が `tauri::async_runtime::spawn` で起動した async task の中で、内部に `std::sync::Mutex::lock()` + 同期 `std::io::Write::write_all` + flush を含む `session.write` を直接呼んでいる。ConPTY 側 buffer が詰まると tokio worker がブロックして他の async タスクが進まなくなる。同等の bracketed paste をやっている `team_hub::inject::inject_once` は意図的に `tokio::task::spawn_blocking` で逃がしているのに、Codex prompt 経路だけパターンが揃っていない。

## 根本原因 / Root cause

- `src-tauri/src/commands/terminal.rs:94-139` の `inject_codex_prompt_to_pty`:
  ```rust
  tauri::async_runtime::spawn(async move {
      tokio::time::sleep(Duration::from_millis(1800)).await;
      let _ = session.write(b"\x1b[200~"); // 同期 blocking write
      // 続けてチャンクを送る
      let _ = session.write(&chunk);
      let _ = session.write(b"\x1b[201~\r");
  });
  ```
  `session.write` 内部は `Mutex::lock()` → `std::io::Write::write_all` → `flush` で完全に同期 blocking。これを tokio runtime の async task 内で直接呼ぶと、ConPTY が back-pressure をかけたとき tokio worker thread を 1 本ずっと占有する。
- 一方 `src-tauri/src/team_hub/inject.rs:347, 392, 423` (`inject_once`) は明示的に:
  ```rust
  let s = session.clone();
  tokio::task::spawn_blocking(move || s.write(&chunk)).await?;
  ```
  と `spawn_blocking` プールに逃がしている (正しい pattern)。
- `inject_codex_prompt_to_pty` だけがこの pattern から外れている。tokio multi-thread runtime のデフォルト worker 数 (= CPU 数) が少ない環境では、4 並列 codex セッションを起動すると runtime が固まる。

## 再現手順 / Repro

1. Canvas で 4 つ以上の codex agent カードを同時に Recruit。
2. 各 codex agent の起動 prompt が長め (10KB 程度) になるよう role profile を整備。
3. 4 並列同時に起動すると、いずれか 1 つの async task の途中で tokio runtime の worker が全消費され、別の `team_*` IPC や fs_watch が一瞬反応しなくなる。
4. ConPTY のバッファが詰まりやすい Windows 環境で再現性が高い。

## 影響範囲 / Impact

- codex 系 agent を多数同時起動するユーザー (`dual-codex-codex` preset 等) に影響。
- runtime の他 task が一時的にブロックされ、UI 全体の反応性が落ちる。
- Windows ConPTY での back-pressure 時に長く露出。

## 修正方針 / Fix proposal

### 推奨: 共通 helper `pty::session::inject_text` を新設して `team_hub::inject` と `inject_codex_prompt_to_pty` を統一

1. `src-tauri/src/pty/session.rs` に async helper を追加:
   ```rust
   impl SessionHandle {
       pub async fn inject_text(self: Arc<Self>, text: &[u8]) -> std::io::Result<()> {
           let s = self.clone();
           let bytes = text.to_vec();
           tokio::task::spawn_blocking(move || {
               // 内部で injecting guard (B-10)、bracketed paste、chunking、Windows 64B/15ms を実装
               s.write_bracketed_paste(&bytes)
           })
           .await
           .map_err(|e| std::io::Error::new(std::io::ErrorKind::Other, e))??;
           Ok(())
       }
   }
   ```
2. `inject_codex_prompt_to_pty`:
   ```rust
   tauri::async_runtime::spawn(async move {
       tokio::time::sleep(Duration::from_millis(1800)).await;
       if let Err(e) = session.inject_text(prompt.as_bytes()).await {
           tracing::warn!(?e, "codex prompt inject failed");
       }
   });
   ```
3. `team_hub::inject::inject_once` も同じ `inject_text` (もしくは内部の `write_bracketed_paste`) に置き換え、code 重複を解消。
4. B-10 の `InjectingGuard` を `inject_text` 内部で必ず張る。

### 短期 minimal
共通化を後回しにする場合、`inject_codex_prompt_to_pty` の `session.write(...)` 各呼び出しを `tokio::task::spawn_blocking(move || session.write(...)).await` に置き換えるだけでも症状は解消する。
