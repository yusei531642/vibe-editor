TITLE: [bug] pty: SessionHandle::drop が Mutex poison で kill を silently スキップする
LABELS: bug,rust,backend

## 概要 / Summary

`SessionHandle::drop` が `Mutex::lock()` の戻り値を `if let Ok(...)` で受けるだけで、poison error の場合は何もせずに drop が抜けてしまう。Issue #144 で「Drop で確実に kill」とした修正意図が、Mutex poison 経路で破綻する。Windows ConPTY は親プロセスが死んでも子が残るケースが多く、kill が silent skip されると orphan child / orphan PTY が leak する。

## 根本原因 / Root cause

- `src-tauri/src/pty/session.rs:187-193`:
  ```rust
  impl Drop for SessionHandle {
      fn drop(&mut self) {
          if let Ok(mut k) = self.killer.lock() {
              let _ = k.kill();
          }
          // poison 時は何も呼ばれない
      }
  }
  ```
- `self.killer` は inject task が panic した場合などに poison しうる。
- `src-tauri/src/pty/registry.rs:47-57` には既に `recover()` ヘルパーが存在し、poison guard を `into_inner()` で剥がして使うパターンが採用されている。Drop だけがこの pattern を取りこぼしている。
- 結果: panic 後の cleanup で kill が silent skip → 子プロセスが残留 → Windows でユーザーが「閉じたはずなのに python.exe / claude.exe が居る」現象。

## 再現手順 / Repro

1. inject task を panic させる (例: 一時的に `panic!` を仕込む or 何らかの 1 in 1000 race)。
2. それを受けた SessionHandle が drop される。
3. Task Manager (Windows) で当該 child process を観察 → kill されず残る。
4. 通常運用では稀だが、長時間稼働で leak が累積。

## 影響範囲 / Impact

- panic / 例外発生時の child process leak。
- Windows で特に顕著 (ConPTY の子プロセスは親死亡後も残る)。
- 累積で system resource 圧迫 → 最終的に PTY 起動失敗。
- ユーザーには「閉じたのに残ってる」と見える低頻度バグ。

## 修正方針 / Fix proposal

### 推奨: `recover()` パターンを Drop でも使う

1. `src-tauri/src/pty/session.rs:187-193` を以下に変更:
   ```rust
   impl Drop for SessionHandle {
       fn drop(&mut self) {
           let mut k = match self.killer.lock() {
               Ok(g) => g,
               Err(p) => {
                   tracing::warn!("SessionHandle: killer mutex poisoned, recovering for kill");
                   p.into_inner()
               }
           };
           if let Err(e) = k.kill() {
               tracing::warn!(?e, "SessionHandle: child kill failed during drop");
           }
       }
   }
   ```
2. 既存の `registry.rs:47-57` の `recover()` ヘルパーがあれば再利用 (パターン統一):
   ```rust
   let mut k = self.killer.recover();
   let _ = k.kill();
   ```
3. (任意) `child` (PTY child handle 自体) も同パターンで poison-safe に。
4. (将来) `SessionHandle` に `Arc<AtomicBool> stopping` を持たせて、drop 中であることを reader thread / inject task に通知して clean shutdown を加速する案 (C-6 と統合可能)。

### 補足
- `into_inner()` は poison guard の中身を取り出すだけで、論理的にデータが破損しているとは限らない (kill する目的なら問題ない)。
- 真の panic-safety を目指すなら `Mutex` ではなく `parking_lot::Mutex` を使う案もあるが、後方互換性のため最小変更で `recover` パターンに留めるのが安全。
