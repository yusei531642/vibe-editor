TITLE: [security] backend: git_diff の path 検証を safe_join 単独に統一 (substring contains 削除)
LABELS: security,bug,rust,backend

## 概要 / Summary

`git_diff` IPC の `original_rel_path` / `rel_path` 検証で、`head_path.contains("..")` のような **substring contains マッチ** を残しているため、`foo..bar` のような無害な path も拒否され、逆に `safe_join` 単独で完結すべき path traversal 検証ロジックが二箇所に分散している。`safe_join` (Component::ParentDir をコンポーネント単位で stack pop) のほうが堅牢かつ一貫性があるので、substring contains 検証は削除して `safe_join` 単独に統一すべき。

## 根本原因 / Root cause

- `src-tauri/src/commands/git.rs:293-304` 付近に以下のような検証が存在:
  ```rust
  if head_path.contains("..") {
      return Err("path traversal".into());
  }
  let abs = safe_join(&project_root, &head_path)?;
  ```
- `head_path.contains("..")` は文字列 substring マッチで:
  - false negative: `..\\` (OS によって) を見落とす (`..` 自体は捕まえるが、組み合わせ次第で抜ける可能性)
  - false positive: `foo..bar.txt` のような連続ドットを含む正当なファイル名を弾く
- `safe_join` (Component::ParentDir をコンポーネントとして処理) のほうが構造的に正しく、true に堅牢。
- 二重検証は単に「異常時の早期 reject」だが、`safe_join` 自体が `Err` を返すため substring contains は冗長 + 誤検知源。

## 再現手順 / Repro

1. `foo..bar.txt` というファイル名 (リテラルに連続ドットを含む正当な名前) を作成。
2. `git_diff({ rel_path: "foo..bar.txt" })` を呼ぶ。
3. substring contains 検証で `..` を含むため `Err("path traversal")` で拒否される (false positive)。
4. 一方、git CLI 自体は `foo..bar.txt` を rev range と解釈する可能性があるため、別の混乱の元にもなる。

## 影響範囲 / Impact

- 連続ドットを含むファイル名 (例: `version 1..2 backup.txt`、稀ではあるが正当) の diff が見れない。
- 本質的な path traversal 防御は `safe_join` で完結しているため、二重検証が混乱を生むだけ。
- 同パターンが他の IPC (files, fs_watch 等) にも残っている可能性があり、構造的な整理対象。

## 修正方針 / Fix proposal

### 推奨: substring contains 検証を削除し、`safe_join` の戻り値だけを真とする

1. `src-tauri/src/commands/git.rs:293-304` の以下を削除:
   ```rust
   if head_path.contains("..") {
       return Err("path traversal".into());
   }
   ```
2. 直後の `safe_join(&project_root, &head_path)?` だけを残す。`safe_join` は `Component::ParentDir` (= `..`) を stack pop でコンポーネント単位に処理し、project_root 配下に閉じ込めない場合は `Err` を返すため、これが path traversal 防御の single source of truth になる。
3. `-` prefix チェック (rev spec disambiguation 防止) は残す:
   ```rust
   if head_path.starts_with('-') {
       return Err("invalid path".into());
   }
   ```
   さらに、git CLI 呼び出し側で `--` separator を必ず挟む形に:
   ```rust
   git_args.push("--");
   git_args.push(&head_path);
   ```
   これにより `foo..bar.txt` のような名前を rev range と誤解釈されない。

### 同パターン横展開
`commands/git.rs` 以外で `path.contains("..")` のような substring 検証が残っていないか grep し、見つけたら一斉に削除する (B-15 の共通 helper への集約とも整合)。
