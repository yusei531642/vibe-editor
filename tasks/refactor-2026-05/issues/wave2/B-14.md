TITLE: [security] backend: paste image を 0o600 に絞り user-only 読み取りに限定
LABELS: security,bug,rust,backend

## 概要 / Summary

`terminal_save_pasted_image` が paste されたクリップボード画像を `~/.vibe-editor/paste-images/paste-{uuid}.png` に書き出すが、ファイル permission を絞っていない (Unix なら `0o644`、umask 依存)。多 user 環境 / shared host では別 user が読み取れる。プロンプト screenshot や機密 UI を含む可能性が高い情報資産なので、handoffs.rs 同様に `restrict_private_file` で `0o600` に絞り、user 自身しか読めなくするのが正しい。

## 根本原因 / Root cause

- `src-tauri/src/commands/terminal/paste_image.rs:54-119` で:
  ```rust
  std::fs::write(&path, &bytes)?;
  // permissions の絞り込みなし
  ```
- 一方、`src-tauri/src/commands/handoffs.rs` 等では `restrict_private_file(&path)` で `0o600` を強制している (Unix 系)。Windows では DACL を user only にする処理。
- paste-images はターミナル経由でユーザーが手動 paste した画像 (= スクリーンショット / プロンプト断片 / 機密 UI 等) を保存するため、外部読み取り可能 permission は明確に privacy 違反。

## 再現手順 / Repro

1. Linux/macOS で multi-user 環境 (例: `chmod 755 ~ && chmod 755 ~/.vibe-editor`) を作る。
2. vibe-editor で画像をクリップボード paste でターミナルに送る。
3. `~/.vibe-editor/paste-images/paste-*.png` を別 user (`other_user`) で `cat` する → 読み取れる (現状)。
4. 期待動作: permission denied (`0o600`)。

## 影響範囲 / Impact

- multi-user host (大学 / 企業の shared dev box / Linux server へ ssh で開く構成) でユーザーの paste 画像が他 user に漏洩。
- Windows でも (低 ACL の設定で) 似たことが起こりうる。
- セキュリティ違反として軽微〜中程度だが、修正コストが極めて低いので必ず潰す。

## 修正方針 / Fix proposal

### 推奨: 書き込み後に `restrict_private_file` で permission を絞る

1. `src-tauri/src/commands/terminal/paste_image.rs` で書き込み後:
   ```rust
   std::fs::write(&path, &bytes)?;
   restrict_private_file(&path)?; // handoffs.rs と同流儀
   ```
   `restrict_private_file` は `src-tauri/src/util/` 等にあれば再利用、なければ:
   ```rust
   #[cfg(unix)]
   fn restrict_private_file(path: &Path) -> std::io::Result<()> {
       use std::os::unix::fs::PermissionsExt;
       let mut perm = std::fs::metadata(path)?.permissions();
       perm.set_mode(0o600);
       std::fs::set_permissions(path, perm)
   }
   #[cfg(windows)]
   fn restrict_private_file(_path: &Path) -> std::io::Result<()> {
       // DACL を user only に絞る (icacls 等) — best-effort
       Ok(())
   }
   ```
2. **書き込みより前に絞る案 (better)**:
   ```rust
   #[cfg(unix)]
   {
       use std::os::unix::fs::OpenOptionsExt;
       let mut f = std::fs::OpenOptions::new()
           .write(true).create(true).truncate(true).mode(0o600)
           .open(&path)?;
       f.write_all(&bytes)?;
   }
   #[cfg(not(unix))]
   { std::fs::write(&path, &bytes)?; }
   ```
   こちらは race 0 で permission を絞れる (より安全)。
3. (将来) `restrict_private_file` は A-10 の `atomic_write(mode: Option<u32>)` の拡張に統合できる (mcp_config / role-profiles も `0o600` 必要)。本 issue は paste-images に focus し、A-10 の helper が用意されたら paste-images もそれを使う形に rewire。
4. (将来) project root 配下の `.vibe-editor/paste-images/` に隔離する案もあるが、後方互換性とリスクが釣り合わないので別 issue (#unassigned) で検討。
