TITLE: [security] backend: IPC 入力サイズ・charset を共通 helper で gate (DoS / log injection 抑止)
LABELS: security,bug,rust,backend

## 概要 / Summary

`recruit_observed_while_hidden` / `team_history_save` / `team_presets_save` などの IPC で、agent_id / team_id / preset_id / serialize された entry の **長さ上限・charset 検証** が局所的でバラバラに書かれている。共通 helper (例: `commands/validation.rs`) を新設し、`is_valid_terminal_id` を一般化した `is_valid_id_segment(s, len_max=64)` / `assert_max_size(serialized, 1MB)` 等の API を整備して全 IPC で使い回す。これにより DoS (巨大な team-history JSON で disk full) / log injection (制御文字混入で tracing log を破壊) を抑止できる。

## 根本原因 / Root cause

- `src-tauri/src/commands/team_state.rs:397-417` (`recruit_observed_while_hidden`):
  - `team_id` / `agent_id` の正規表現検証なし。任意 string が project_key/safe_segment に流れ、log にも記録される。
- `src-tauri/src/commands/team_history.rs:204-224` (`team_history_save`):
  - `entry` 全体の serialized サイズ上限なし。renderer から悪意ある呼び出しで巨大 JSON を吐ける (disk full の DoS 経路)。
- `commands/validation.rs` 相当が現状存在せず、`is_valid_terminal_id` (terminal id 用) だけが個別に定義されている (`commands/terminal.rs` 内)。
- 各 IPC が必要に応じて手書き検証するため、抜け漏れの一覧化ができない。

## 再現手順 / Repro

### 巨大 entry での DoS
1. renderer 側で `invoke('team_history_save', { entry: { ...大量の garbage 100MB... } })` を呼ぶ。
2. Rust 側に長さ上限がないため、そのまま `~/.vibe-editor/team-history/<encoded>/...json` に書き出される。
3. disk full まで反復可能。

### Log injection
1. `recruit_observed_while_hidden({ team_id: "evil\nINFO impersonated_log_line", agent_id: "..." })` を呼ぶ。
2. Rust 側が `tracing::info!(team_id = %team_id, ...)` で記録すると、改行を含む string が log に流れて改竄ログを混入できる。

## 影響範囲 / Impact

- DoS: renderer (もしくは renderer 経由の悪意ある preload script) から disk を埋め尽くす経路。
- Log injection: ログ改竄 / 監査痕跡撹乱。
- 横断的な検証が分散しており、新規 IPC を追加するたびに同種ミスを誘発しやすい。

## 修正方針 / Fix proposal

### 推奨: `commands/validation.rs` を新設し共通 helper を用意

1. `src-tauri/src/commands/validation.rs` を新設:
   ```rust
   use std::sync::OnceLock;
   use regex::Regex;

   static ID_SEGMENT_RE: OnceLock<Regex> = OnceLock::new();

   pub fn is_valid_id_segment(s: &str) -> bool {
       let re = ID_SEGMENT_RE.get_or_init(|| Regex::new(r"^[A-Za-z0-9_\-]{1,64}$").unwrap());
       re.is_match(s)
   }

   pub fn validate_id_segment<'a>(name: &str, s: &'a str) -> Result<&'a str, String> {
       if !is_valid_id_segment(s) {
           return Err(format!("invalid {}: must match [A-Za-z0-9_-]{{1,64}}", name));
       }
       Ok(s)
   }

   pub const MAX_PERSIST_PAYLOAD: usize = 1 * 1024 * 1024; // 1 MiB

   pub fn assert_max_size(payload_size: usize, max: usize) -> Result<(), String> {
       if payload_size > max {
           return Err(format!("payload too large: {} > {}", payload_size, max));
       }
       Ok(())
   }

   pub fn is_valid_terminal_id(s: &str) -> bool { is_valid_id_segment(s) }

   /// 制御文字 / 改行を除去 (log injection 防御)
   pub fn sanitize_for_log(s: &str, max_len: usize) -> String {
       s.chars()
           .filter(|c| !c.is_control())
           .take(max_len)
           .collect()
   }
   ```
2. `commands/mod.rs` に `pub mod validation;` を追加 (tauri-ipc-commands skill の 5 点同期チェック)。
3. 既存 IPC を順次移行:
   - `terminal.rs::is_valid_terminal_id` → `validation::is_valid_terminal_id` の wrapper / re-export。
   - `team_state.rs::recruit_observed_while_hidden`:
     ```rust
     validation::validate_id_segment("team_id", &team_id)?;
     validation::validate_id_segment("agent_id", &agent_id)?;
     ```
   - `team_history.rs::team_history_save`:
     ```rust
     let serialized = serde_json::to_vec(&entry)?;
     validation::assert_max_size(serialized.len(), validation::MAX_PERSIST_PAYLOAD)?;
     ```
   - `team_presets.rs::team_presets_save` も同様。
   - `terminal.rs::terminal_kill / resize / write` で `id` を `validate_id_segment` に通す (既存 issue / Tier D の指摘とも統合)。
4. tracing log で user input を出すときは `sanitize_for_log` を経由:
   ```rust
   tracing::info!(team_id = %validation::sanitize_for_log(&team_id, 64), ...);
   ```

### 短期 minimal
本 issue では `validation.rs` 新設 + `recruit_observed_while_hidden` / `team_history_save` / `team_presets_save` の 3 IPC への適用を必須スコープとする。残りの IPC (terminal_kill / resize / write 等) は本 PR の余力があれば、なければ後続 issue に分けて移行。
