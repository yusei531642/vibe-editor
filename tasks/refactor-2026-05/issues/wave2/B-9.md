TITLE: [bug] pty: Windows ConPTY 出力が CP932 シェルで U+FFFD 化する (#120 を batcher に拡張)
LABELS: bug,rust,backend

## 概要 / Summary

PTY reader が読み取った生バイトを `String::from_utf8_lossy` でそのまま UTF-8 として解釈しているため、Windows console 既定 OEM コードページ (CP932 / ja-JP) で動くシェル / `python -c "print('日本語')"` / `git log` の native error / `dir` の漢字ファイル名出力が全文字 U+FFFD (replacement character) に置換される。Issue #120 で導入した CP932 / Shift_JIS デコードはファイル読み出し経路にしか適用されておらず、PTY batcher / scrollback には射程外。

## 根本原因 / Root cause

- `src-tauri/src/pty/batcher.rs:119-147` の `extract_emit_payload`:
  ```rust
  let s = String::from_utf8_lossy(&buf).to_string();
  ```
  生バイトを UTF-8 として解釈、不正バイトは U+FFFD に置換。
- `src-tauri/src/pty/scrollback.rs:46-82` の `scrollback_to_string` も同じく `String::from_utf8_lossy`。
- 既存の Shift_JIS デコーダー (`src-tauri/src/commands/files/encoding.rs:48-49` の `SHIFT_JIS.decode`) は files 経路にしか使われていない。
- Windows ConPTY は親プロセスが UTF-8 を強制しない限り、シェルがロケール依存で書き出すバイト列を passthrough する。日本語環境では CP932 になりがち。

## 再現手順 / Repro

1. Windows 11 で vibe-editor を起動。
2. 内蔵 terminal で `chcp 932` を実行 (または何もしない初期 cmd.exe)。
3. `dir` を実行し、漢字ファイル名 (例: `テスト.txt`) のあるディレクトリを表示。
4. ファイル名の漢字部分が全て `?` または白四角 (U+FFFD) で表示される。
5. `python -c "print('こんにちは')"` でも同様に文字化け。

## 影響範囲 / Impact

- Windows + 日本語ロケールの全ユーザーに影響 (vibe-editor のターゲット層と重なる)。
- 既存ターミナルでの作業 (git log の native error, dir の漢字ファイル名, python script 出力) が読めない。
- Issue #120 で files に CP932 対応したが、ターミナル経路には届いていなかった (#120 の補完)。

## 修正方針 / Fix proposal

### 案 (a): Rust 側 batcher に encoding pipeline を追加
1. PTY 起動時にシェルの codepage を判定 (Windows: `GetConsoleOutputCP()`、または起動時 PowerShell に `[Console]::OutputEncoding.CodePage` を一発問い合わせ)。
2. Codepage が 932 (CP932) の場合、batcher の `extract_emit_payload` で `encoding_rs::SHIFT_JIS.decode_*_with_replacement` を適用。
3. **重要**: chunk 境界で multi-byte 途中 split が発生する。decoder state を `SessionHandle` に持たせて chunk をまたいで継続させる:
   ```rust
   pub struct SessionHandle {
       decoder: Option<Mutex<encoding_rs::Decoder>>, // Codepage に応じて初期化
       ...
   }
   // batcher 側
   let mut out = String::with_capacity(buf.len() * 2);
   decoder.decode_to_string(&buf, &mut out, last); // last = false で state 継続
   ```
4. scrollback 側も同じ decoder state を流用。
5. UTF-8 シェルでは `decoder = None` で fast path (`from_utf8_lossy`)。

**短所**: 状態管理が増え、reader thread の lock 競合が複雑化する。

### 案 (b) (推奨): PowerShell / cmd 起動時に `chcp 65001` を初期コマンドとして強制注入し UTF-8 化を保証
1. `src-tauri/src/pty/session.rs` (もしくは `commands/terminal.rs::create`) で Windows ConPTY 起動直後に bracketed-paste なしで `chcp 65001\r` を inject (ユーザー入力前に内部で先行送信)。
2. PowerShell 起動の場合は `[Console]::OutputEncoding = [System.Text.UTF8Encoding]::new(); chcp 65001\r` を inject。
3. xterm.js 側は最初の prompt が出るまで `chcp 65001` の echo を hide (もしくは prompt 検出で削除) する。

**短所**: chcp の echo がユーザーの目に一瞬触れる可能性がある (許容範囲)。代替シェル (nu, fish, etc.) では効かない (代替シェルは UTF-8 既定の場合が多く、問題が出にくい)。

**長所**:
- pipeline 改造が不要、後方互換性が高い。
- すべての出力が UTF-8 になるので scrollback / batcher の既存ロジックがそのまま動く。
- WSL / git bash 等の non-Windows シェルでは `chcp` がエラーを出すだけで実害なし (検出して skip 可能)。

### 推奨実装順
1. まず案 (b) を入れて症状を解消する (短期 fix)。
2. 案 (a) は将来の roadmap (PTY pipeline 抽象化、Tier D の refactor opportunity) として入れる。`SHIFT_JIS` 以外の codepage (CP949, CP936, CP1252) もサポートする際にやる。
