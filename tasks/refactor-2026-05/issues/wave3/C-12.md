TITLE: [security] spool_long_payload: project_root 検証なし、canonicalize 失敗時の素 path フォールバック
LABELS: security,rust,backend

## 概要 / Summary

TeamHub の `spool_long_payload` は team_send 等で大きな payload を一時ファイルに退避して worker に file path として渡す機構だが、入口で `project_root` 引数の検証 (絶対 path / `..` 検出 / canonicalize 必須) を行っていない。canonicalize が失敗 (= 不存在 / 権限不足) した場合に raw path をそのまま使う fallback があるため、`..` traversal や symlink 経由で意図しない場所に書ける余地が残る。

## 根本原因 / Root cause

- `src-tauri/src/team_hub/spool.rs:54-91`
- `src-tauri/src/team_hub/state.rs:1150-1242` (caller 経路)
- `spool_long_payload(project_root, payload)` で:
  - `Path::new(project_root).is_absolute()` 確認なし
  - `project_root.contains("..")` 等の traversal check なし
  - `canonicalize` 失敗時に `project_root` を素のまま使う fallback (= raw path で書く) が残っている
- 結果、attacker が制御する worker から `team_send({ data: <大きな payload>, project_root: "../../etc/.../" })` を送れば、spool が想定外のディレクトリに書かれる可能性。
- 同 user 権限内に限定されるとはいえ、canonicalize bypass は他の hardening と整合しない。

## 再現手順 / Repro

1. 攻撃 worker が JSON-RPC 経由で TeamHub に直接接続 (token を盗んでいる前提だが、A-5 で塞がれる前の状況)。
2. `team_send({ data: <large>, project_root: "/tmp/../home/<user>/Desktop" })` 等を送る。
3. `spool_long_payload` の canonicalize がたまたま失敗するパスを巧妙に構築すると、想定外の Desktop 配下に payload が書ける。

## 影響範囲 / Impact

- TeamHub に到達できる worker (token 必要なので前提条件あり)
- Token 盗難経路 (A-5) と組み合わせると同一ホスト内の他ユーザー (Linux multi-user setup) への波及あり
- severity: MEDIUM (security / path traversal、攻撃成立のハードルは中程度)

## 修正方針 / Fix proposal

### 案1 (推奨): 入口で絶対 path + canonicalize 必須化

```rust
fn validate_spool_root(project_root: &str) -> Result<PathBuf, ToolError> {
    let p = Path::new(project_root);
    if !p.is_absolute() {
        return Err(ToolError::invalid_args("project_root must be absolute"));
    }
    if p.components().any(|c| matches!(c, Component::ParentDir)) {
        return Err(ToolError::invalid_args("project_root must not contain .."));
    }
    p.canonicalize().map_err(|e| ToolError::invalid_args(format!("project_root canonicalize failed: {e}")))
}
```

`spool_long_payload` の冒頭で必ず通す。canonicalize 失敗時の fallback を **削除**。

### 案2: active project_root と一致確認を導入

Hub state に「現在 active な project_root の canonical」を保持し、`spool_long_payload` の `project_root` 引数とそれが一致しなければ拒否。team_state_read 等で他の Tier A 修正で導入する `assert_active_project_root!` helper を使い回す。

### 案3 (補強): spool ディレクトリ自体を `vibe_root() / "spool"` に固定し、project_root 引数を取らない

そもそも `project_root` を引数に取らず、固定の `~/.vibe-editor/spool/<team_id>/<uuid>` に書く。caller 側に project_root を意識させない。
