TITLE: [refactor] agent_role_bindings: team_id 次元を持たないため cross-team で role 上書きの余地
LABELS: refactor,rust,backend

## 概要 / Summary

TeamHub の `agent_role_bindings` Map は handshake 時に「`agent_id -> role`」のキーで role を固定するが、team_id 次元を持たない単一キーなので、同じ `agent_id` が別 team で再 handshake すると古い binding が上書きされる。複数 team を持つシナリオで cross-team の role 漏れ / 古い role での処理続行 race が成立する。

## 根本原因 / Root cause

- `src-tauri/src/team_hub/state.rs:798-860`
- `agent_role_bindings: HashMap<String, String>` のキーが `agent_id` 単独。
- handshake で「同 agent_id が別 team から再接続」した場合、既存 bind を replace するため、古い接続が in-flight で role を参照すると更新済みの role を見てしまう。
- multi-team / dual preset シナリオで 1 worker pool を複数 team が使い回すパターンでは、cross-team の role 上書きが成立する。

## 再現手順 / Repro

1. team A を leader で立てる。worker `agent-1` (role=`programmer`) が join。
2. team B を別途立てる。同じ `agent-1` を異なる role (`reviewer`) で join させる (実装上は別プロセスでも agent_id が衝突しうる)。
3. team A の文脈で `agent_role_bindings.get("agent-1")` を引くと `reviewer` が返り、本来 team A 内では `programmer` だったはずの role 判定が壊れる。
4. permission check / `assert_task_owner_or_leader` 等で誤った role と照合し、authz の動作が不安定になる。

## 影響範囲 / Impact

- multi-team / dual preset 利用者 (workspace-presets の `dual-claude-claude` 等)
- agent_id を team 跨ぎで使い回す設計を持つ将来の機能
- severity: MEDIUM (refactor / 認可 race の温床)

## 修正方針 / Fix proposal

### 案1 (推奨): キーを (agent_id, team_id) tuple にする

```rust
// before
agent_role_bindings: HashMap<String, String>,
// after
agent_role_bindings: HashMap<(String, String), String>, // (agent_id, team_id) -> role
```

- handshake 時に `team_id` を必ず受け取り tuple key で insert。
- `team_dismiss` 経路でも `(agent_id, team_id)` で remove する。
- 既存 callsite を一括書き換え。`team_id` が未取得の経路は handshake のフローで取得を強制 (Hub 入口は token + team_id がペアで届くはず)。

### 案2: Map<String, Map<String, String>> の入れ子化

`team_id -> (agent_id -> role)` という二段構造。case 別に lookup を切り替える callsite が読みやすくなるが、tuple key の方が衝突なくシンプル。

### 補強: persistence (`role-profiles.json#dynamic[]`) も同形に揃える

現在 `dynamic` 永続化部も agent_id 単独であれば、team_id を含めた形に migration を入れる (schema_version bump)。
