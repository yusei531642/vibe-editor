TITLE: [bug] file_locks: socket 異常切断で lock が残留、自動解放経路が dismiss MCP 呼び出しのみ
LABELS: bug,rust,backend

## 概要 / Summary

TeamHub の `file_locks` 機構は `team_dismiss` MCP の呼び出しでのみ lock を解放する設計だが、worker が socket / named pipe を異常切断した場合 (process crash / network drop / 意図的 kill) には dismiss が呼ばれず、lock がプロセス再起動まで残る。Hub 再起動を跨ぐ永続化も無いため、stale lock が他 worker の作業を block する hardening 不足。

## 根本原因 / Root cause

- `src-tauri/src/team_hub/protocol/tools/dismiss.rs:64-79`
- `src-tauri/src/team_hub/protocol/tools/assign_task.rs:256-269`
- 解放経路は `team_dismiss` の MCP tool 呼び出し時のみ。
- socket 切断 (`handle_client` の return path) では `agent_id` の lock を一括解放する hook が無い。
- 結果として:
  - worker process crash → dismiss されないまま socket close → lock 残留
  - 同じ agent_id で worker を再 spawn しても、新しい接続では「自分が前回取った lock」かどうか分からないため、lock が gridlock になる
- `assign_task` の lock peek が古いスナップショットを参照する race も併発しうる。

## 再現手順 / Repro

1. team を recruit し worker A が `team_lock_files({ paths: ["src/foo.rs"] })` で lock を取る。
2. worker A の process を `kill -9` する (crash 模倣)。
3. Hub 側に socket 切断は伝わるが、`file_locks` 内の `src/foo.rs` lock は残ったまま。
4. 別 worker B が `team_lock_files({ paths: ["src/foo.rs"] })` を呼ぶと拒否される (dead worker A の lock が居座り)。
5. team を dismiss しないと永続的に lock が解けない。

## 影響範囲 / Impact

- worker が crash しやすい開発時 (network 不安定 / 強制 kill) の作業ブロック
- multi-worker チーム作業の停止リスク
- severity: MEDIUM (bug、回避策 = team dismiss + 再 recruit はあるがコスト高)

## 修正方針 / Fix proposal

### 案1 (推奨): socket 切断 hook で agent_id の lock を一括解放

1. `team_hub::handle_client` の終了 path (Ok / Err 双方) で `state.lock().await.release_locks_for_agent(&agent_id)` を呼ぶ。
2. release 関数:
   ```rust
   pub fn release_locks_for_agent(&mut self, agent_id: &str) {
       self.file_locks.retain(|_path, lock| lock.held_by != agent_id);
   }
   ```
3. `team_dismiss` 経路でも同じ helper を使い、二重定義を避ける。

### 案2: lock に TTL を入れて N 分自動失効

worker が live keepalive を送らない場合 N 分経過で自動失効。実装は単純だが、長時間 lock を保持したい legitimate ユースを誤殺する可能性あり。案1 と組み合わせると最も robust。

### 案3: lock の persistence

Hub 再起動を跨ぐ orphan を防ぐため、`file_locks` を `~/.vibe-editor/file-locks/<team>.json` に persist。Hub 起動時に load し、現存 worker の agent_id にマッチしないものを起動時 cleanup。Wave 3 の範囲外で、Tier D roadmap として記録。
