TITLE: [security] backend: app_set_project_root に is_safe_watch_root と同水準の path 検証を導入
LABELS: security,bug,rust,backend

## 概要 / Summary

`app_set_project_root` は dialog (`dialog_open_folder`) で取得した path を renderer 経由で受け取って active project root を切り替えるが、入口で path が「安全に watch できるか」「ホーム配下か」「システム denylist (例えば `/`、`/etc`、`C:\Windows` 等) に該当しないか」の検証を行っていない。改ざん済み JS から `app_set_project_root("/etc")` を直接 invoke すれば、後続の `fs_watch` / `git_*` IPC が信頼できない場所で発火する TOCTOU 攻撃が成立する。

## 根本原因 / Root cause

- `src-tauri/src/commands/dialog.rs:9-33`: dialog の戻り値が後続 IPC で再検証されない設計。
- `app_set_project_root` (`src-tauri/src/commands/app/window.rs` 系) で path validation が局所的 / ザル。
- 一方 `fs_watch::is_safe_watch_root` (`src-tauri/src/commands/fs_watch.rs`) は denylist + ホーム配下チェックを実装済み。同水準のチェックを project_root setter に流用していないのが矛盾。
- `dialog_open_folder` は OS picker を経由するから安全と暗黙に仮定されているが、改ざん済み renderer から直接 IPC 呼び出しが可能なため、信頼できない。

## 再現手順 / Repro

1. DevTools (or 改ざん済み bundle) から `window.api.app_set_project_root({ projectRoot: '/etc' })` を invoke。
2. active project root が `/etc` に切り替わる。
3. 続けて `window.api.git_status({ projectRoot: '/etc' })` を呼ぶと、`/etc` を git repo として走査しエラーが出る (もしくは `/etc/.git` がたまたま存在すれば内容が leak)。
4. `fs_watch::start_for_root('/etc')` も後段で発火する可能性。

## 影響範囲 / Impact

- 改ざん bundle / XSS 経由の攻撃者が IPC を直接呼べる前提では、project_root 切替で system 領域への走査ができてしまう
- 通常のユーザー操作だけでは表面化しないが、defense-in-depth が崩れている
- severity: MEDIUM (security / hardening)

## 修正方針 / Fix proposal

### 案1 (推奨): is_safe_watch_root と同等の検証を共通 helper 化して app_set_project_root から呼ぶ

1. `src-tauri/src/commands/path_guard.rs` (新規) もしくは既存 util に `pub fn validate_project_root(path: &Path) -> Result<PathBuf, CommandError>` を作る。
   - `path.canonicalize()` 必須
   - `is_absolute()` 必須
   - ホーム配下 or 既知の dev directory (例: `/Users/*/repos`, `C:\Users\*\Documents`) のいずれかに該当
   - denylist (`/etc`, `/var`, `/sys`, `C:\Windows`, `C:\Program Files` etc.) を reject
2. `app_set_project_root` 入口で `validate_project_root(&pr)?` を呼ぶ。失敗時は `CommandError::InvalidPath` で renderer に明示的なエラーを返す。
3. `fs_watch::is_safe_watch_root` も同 helper に置換 (DRY 化)。
4. OnboardingWizard `chooseFolder` 経路でも IPC 引数として `dialog_validate_project_root` (validation のみ) を呼べるようにし、ピックフォルダ即時拒否を可能に。

### 案2: dialog_open_folder の戻り値に署名を付ける

dialog で取得したことを示す HMAC 署名を Rust 側で生成し、後続 IPC が署名を検証して「dialog 経由かどうか」を判別。複雑だが TOCTOU を本質的に塞げる。Wave 3 では案1 推奨。
