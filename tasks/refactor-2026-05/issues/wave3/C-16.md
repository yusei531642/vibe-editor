TITLE: [bug] backend: team_history_save の disk write 失敗時に cache が rollback されない
LABELS: bug,rust,backend,persistence

## 概要 / Summary

`team_history_save` は in-memory cache を更新してから disk に `save_all` を呼ぶ順序で動くが、disk write が失敗した場合に cache が rollback されないため、cache (= renderer 側の真) と disk の中身が乖離した状態が残る。次回起動時に disk から load された古い state が使われ、ユーザーから見ると「セッション履歴が一部消えた」ように見える。

## 根本原因 / Root cause

- `src-tauri/src/commands/team_history.rs:204-224`
- 流れ:
  1. cache に新 entry を `push` / `update` (in-memory)
  2. disk に `save_all()` を呼ぶ
  3. save_all 失敗 → `Err` を返すが、cache は新規状態のまま
- 結果として:
  - renderer は IPC で Err を受けるが、cache は更新済み (次回 IPC `team_history_load` は新 state を返す)
  - プロセス再起動すると disk からは古い state が load される
  - ユーザーは「保存できなかったはず」のつもりが renderer 上では新規データが見えている → 起動後に消える、という UX

## 再現手順 / Repro

1. `~/.vibe-editor/team-history.json` を read-only に chmod する (ENV 制約 / 権限不足の模倣)。
2. vibe-editor で team を recruit して何らかの操作で team_history_save を起こす (orchestration summary 更新等)。
3. IPC は失敗するが、Canvas / TeamDashboard 上では新しい entry が見える。
4. vibe-editor を再起動すると entry が消えている。

## 影響範囲 / Impact

- ファイル権限が読み取り専用の環境 (sandbox / 読み取り専用ホーム) ユーザー
- ディスクフル / disk error が発生したユーザー
- severity: MEDIUM (data consistency bug、頻度は低いが起きると UX 不良)

## 修正方針 / Fix proposal

### 案1 (推奨): cache 更新前に snapshot を取り、save_all 失敗時に rollback

```rust
let snapshot = cache.clone();
mutate_cache(&mut cache, &request);
match save_all(&cache).await {
    Ok(()) => Ok(()),
    Err(e) => {
        cache = snapshot; // rollback
        Err(CommandError::IO(e))
    }
}
```

`Vec::clone()` 1 回のコストを払うが、history は 20 entry/project の上限があり許容範囲。

### 案2: cache を後置更新に変更 (write-through)

```rust
let new_state = mutate_clone(&cache, &request);
save_all(&new_state).await?;
cache = new_state;
```

failure path で cache 更新自体が起きないので rollback 不要。順序を逆にするだけで OK。実装コストが小さく推奨度が高い。

### 補強: `SAVE_LOCK` パターンと組み合わせる

settings/role_profiles/team_history/team_presets で 4 重定義されている `SAVE_LOCK` (Tier D の roadmap 項目) と組み合わせ、save_all を atomic_write 側 helper で wrap する設計に進化させる方向。
