TITLE: [refactor] backend: settings_save に schema_version 互換性ガードを追加
LABELS: refactor,rust,backend,persistence,settings

## 概要 / Summary

`settings_save` IPC は受け取った settings オブジェクトをそのまま disk に書き戻す設計だが、renderer 側のバージョン (古いビルドが残っている / 別 vibe-editor インスタンスが旧 schema で書き戻す) と最新 schema の不一致を検出する手段が無い。古い renderer から新しい schema を **逆方向 downgrade** で上書きする経路があり、最新フィールドが silent 損失する。

## 根本原因 / Root cause

- `src-tauri/src/commands/settings.rs:251-261`
- `settings_save(request: AppSettings)` が `schema_version` を受け取りはするが、保存時に「現行 `APP_SETTINGS_SCHEMA_VERSION` 以上か」をチェックしていない。
- 古い vibe-editor 起動 → 新しい設定ファイルを読む → migration 不能 → 旧 schema の subset で再保存 → 新規追加フィールドが上書きで消える。
- 複数インスタンス (例: portable build と installed build) を行き来すると静かにデータが減る。

## 再現手順 / Repro

1. vibe-editor v1.4.0 (旧 schema) と v1.5.0 (新 schema) を交互に起動。
2. v1.5.0 で新規 setting (例: `glassRadius`) を変更 → 保存。
3. v1.4.0 を起動 → 設定モーダルを開く → 何らかの設定を変更 → 保存。
4. v1.4.0 は `glassRadius` を知らないので serialize 時に欠落 → disk から消える。
5. 次回 v1.5.0 起動時 `glassRadius` がデフォルトに戻っている。

## 影響範囲 / Impact

- 複数 vibe-editor バージョン / portable build を併用するユーザー
- リリース直後の混在期 (auto-update 中の遷移 window)
- severity: MEDIUM (refactor、データ消失リスク)

## 修正方針 / Fix proposal

### 案1 (推奨): schema_version でガードして古い書き戻しを reject

```rust
const APP_SETTINGS_SCHEMA_VERSION: u32 = 5; // 例

if request.schema_version < APP_SETTINGS_SCHEMA_VERSION {
    tracing::warn!(
        client_version = request.schema_version,
        current_version = APP_SETTINGS_SCHEMA_VERSION,
        "settings_save rejected: stale schema",
    );
    return Err(CommandError::SchemaMismatch {
        expected: APP_SETTINGS_SCHEMA_VERSION,
        got: request.schema_version,
    });
}
```

### 案2: merge save (既存 disk 値とマージ)

`settings_save` 受信前に disk から最新 settings を再 load し、未知フィールド (= 受信側に無いフィールド) は disk の値を保持して merge。古い build からの書き戻しでも新規フィールドが消えない。実装は重いが堅牢。

### 補強: settings migration インフラの足場

Rust 側に schema_version ごとの migration 関数を持たせる (`fn migrate_v3_to_v4(v: V3) -> V4`)。Tier D roadmap で「settings migration インフラ」として記載済みのものに着手する第一歩。

## Done criteria
