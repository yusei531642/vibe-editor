TITLE: [bug] team-history: 手編集と並行する auto-save で外部変更がロストする (mtime 検知)
LABELS: bug,rust,backend,persistence

## 概要 / Summary

`team_history.json` の in-memory cache はプロセス内 only で、外部 (他 vibe-editor インスタンス / ユーザーの手編集 / `jq` でのスクリプト編集) との同期がない。auto-save が走るタイミングで外部変更を上書きするため、ユーザーが直前に書いた手編集がロストする stale-write が発生する。

## 根本原因 / Root cause

- `src-tauri/src/commands/team_history.rs:14-17, 134-145`
- 起動時に 1 回だけ disk から load → in-memory cache が真として動く
- `team_history_save` で cache → disk に書く
- その間に他のプロセス (ユーザー or 別インスタンス) が disk を書き換えても検知しない
- 結果: 古い cache の内容で disk を上書きし、外部変更が消える

## 再現手順 / Repro

1. vibe-editor を起動して team を recruit (in-memory cache が初期化される)。
2. 別のエディタで `~/.vibe-editor/team-history.json` を直接編集し、ある entry の `summary` を書き換えて保存。
3. vibe-editor 内で別の team_history_save を起こす操作を行う (orchestration summary 更新等)。
4. disk を再確認すると、手編集した `summary` が cache 由来の旧値で上書きされている。

## 影響範囲 / Impact

- 手編集ユーザー / スクリプトで履歴を加工するパワーユーザー
- 複数 vibe-editor インスタンスを同時起動するユーザー
- severity: MEDIUM (data loss、頻度は低いが起きると痛い)

## 修正方針 / Fix proposal

### 案1 (推奨): save 直前に mtime を再 check し、外部変更があれば reload + merge

```rust
async fn save_with_mtime_guard(cache: &mut Cache, path: &Path) -> Result<()> {
    let metadata = tokio::fs::metadata(path).await?;
    let disk_mtime = metadata.modified()?;
    if disk_mtime > cache.loaded_mtime {
        // 外部変更を検知 → reload + merge
        let disk = load_from_disk(path).await?;
        cache.merge_external(disk);
        cache.loaded_mtime = disk_mtime;
    }
    save_all(cache, path).await?;
    cache.loaded_mtime = tokio::fs::metadata(path).await?.modified()?;
    Ok(())
}
```

merge ロジックは「disk 側の entry id が cache に無ければ追加 / 同 id は cache 側を優先」など、entry-level の semantics を決める必要あり。

### 案2: notify watcher で外部変更を即時検知

`notify` crate を使い、起動時に `team_history.json` を watch。外部書き込みイベントで cache を reload。実装が複雑だが体験は最も robust。

### 案3 (簡易): save 直前に「disk の hash が起動時 hash と一致するか」を確認し、不一致なら abort + ユーザーに通知

「外部変更を検知しました。再起動してください」みたいな toast を出す。merge を実装しない代わりに stale-write を完全に防ぐ。
