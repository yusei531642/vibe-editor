TITLE: [refactor] logging: vibe-editor.log を日次回転 + 古い世代の自動削除 (DoS 経路を塞ぐ)
LABELS: refactor,rust,backend

## 概要 / Summary

vibe-editor の tracing log は `tracing-appender::rolling::never` で `vibe-editor.log` に **無回転** で追記される設計。長期稼働 (数日 / 数週間) で disk を消費し続けるため、ユーザー側で disk full → vibe-editor 自身および同マシンの他アプリが書き込み不能になる DoS 経路が成立する。tracing-appender は標準で `daily` rotation API を持つので、これに切り替えて N 日経過した log を自動削除する pruner を併設する。

## 根本原因 / Root cause

- `src-tauri/src/lib.rs:54` 付近で `tracing_appender::rolling::never(log_dir, "vibe-editor.log")` を使用。
- 無回転のため、長期稼働で `vibe-editor.log` が GB 単位に肥大化する可能性。
- TeamHub / PTY / IPC の活発なログ ($trace`, `debug`) を有効化していると 1 日数百 MB に達することもある。
- Windows / macOS でも tempdir の容量を圧迫し、結果的に他アプリへ波及。

## 再現手順 / Repro

1. vibe-editor を `RUST_LOG=trace` で起動し、Canvas で team を多数 recruit して 24 時間放置。
2. `~/.vibe-editor/logs/vibe-editor.log` のサイズを確認すると数百 MB - GB 単位になっている。
3. これを 1 週間続けると disk 空き容量に visible な打撃が出る。

## 影響範囲 / Impact

- 長期稼働サーバー / 開発機
- log を有効化したまま放置するパワーユーザー
- 一般ユーザーは `info` level でも数日で 100MB を超えうる
- severity: MEDIUM (refactor / availability)

## 修正方針 / Fix proposal

### 案1 (推奨): tracing_appender::rolling::daily に切替 + 古い世代の自動削除 pruner

```rust
let appender = tracing_appender::rolling::daily(&log_dir, "vibe-editor.log");
// ファイル名は vibe-editor.log.2026-05-09 のように日付付きで rotate される
```

加えて起動時に「N 日以上前の log file を best-effort で削除」する pruner を追加:

```rust
fn prune_old_logs(log_dir: &Path, keep_days: u32) {
    let cutoff = SystemTime::now() - Duration::from_secs(60 * 60 * 24 * keep_days as u64);
    if let Ok(entries) = fs::read_dir(log_dir) {
        for e in entries.flatten() {
            let name = e.file_name();
            let s = name.to_string_lossy();
            if !s.starts_with("vibe-editor.log") { continue; }
            if let Ok(meta) = e.metadata() {
                if let Ok(modified) = meta.modified() {
                    if modified < cutoff {
                        let _ = fs::remove_file(e.path());
                    }
                }
            }
        }
    }
}
```

`keep_days` は 14 日 (or settings から取れるように)。

### 案2: size-based rotation (`rolling::Builder::max_log_files`)

`tracing-appender` 0.2.x の builder API で `.max_log_files(14)` を指定。日次 rotation 自体はそのまま、古い世代を自動的に GC させる。最も簡単な実装。

### 補強: log level を settings で制御可能に

ユーザーが Settings から `Log Level` を選べるように。production 環境のデフォルトは `info` (現状) を維持。
