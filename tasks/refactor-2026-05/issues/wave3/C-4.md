TITLE: [bug] canvas: DiffCard / ChangesCard.refresh が unmount 後 setState する
LABELS: bug,canvas,javascript

## 概要 / Summary

`DiffCard` / `ChangesCard` の `refresh` callback (= `useFilesChanged` 等から呼ばれる手動再取得) に cancel フラグが無いため、ユーザーが refresh を押した直後にカードを閉じる / Canvas を clear するなどして unmount すると、await 完了後に setState が走り React の "Can't perform a state update on an unmounted component" 警告 (および稀に memory leak) が発生する。

## 根本原因 / Root cause

- `src/renderer/src/components/canvas/cards/DiffCard.tsx:29-42`
- `src/renderer/src/components/canvas/cards/ChangesCard.tsx:30-38`
- 初回 `useEffect` での fetch は `let cancelled = false; ...; return () => { cancelled = true; };` で cleanup を仕込んで保護している。
- しかし `refresh` 関数 (callback) には cancel フラグが無く、await 完了後に `setItems(...)` / `setOriginal(...)` 等を無条件に呼ぶ。
- unmount 後だとこの setState は捨てられるが、React 18+ では dev mode で warning が出るうえ、頻繁に呼ぶと clear→ unmount のタイミングでエラーログが汚染される。

## 再現手順 / Repro

1. Canvas に DiffCard (or ChangesCard) を 1 枚配置し、git に変更を適用しておく。
2. カード上の「再読み込み」ボタンを押した直後に、200ms 以内にカードの × ボタン (またはキャンバス Clear) でカードを閉じる。
3. Chrome DevTools Console に React の warning が出る。React DevTools Profiler でも unmount 後 commit が観測される。

## 影響範囲 / Impact

- DiffCard / ChangesCard を頻繁に扱う開発者ユーザー (review 用途)
- 機能上の致命的破綻は無いが、warning ノイズで他のエラーが埋もれる
- React 19 では unmount 後 setState が strict-mode 中に明確な error になる可能性
- severity: MEDIUM

## 修正方針 / Fix proposal

### 案1 (推奨): mounted ref を共有する

```ts
const mountedRef = useRef(true);
useEffect(() => () => { mountedRef.current = false; }, []);

const refresh = useCallback(async () => {
  const data = await invoke('git_diff', ...);
  if (!mountedRef.current) return;
  setItems(data.items);
}, []);
```

両 component で同じパターンを使う。

### 案2: AbortController を渡す

`invoke` 経路は AbortController を直接受けないので、wrapper で `if (signal.aborted) return` を挟む。ただし vibe-editor の `tauri-api.ts` 全体を AbortController 対応にする工事が大きいので案1 推奨。

### 案3: useFilesChanged 側で cancel を一括管理

`useFilesChanged` hook 内で cancel フラグを持ち、refresh が再入したら前回をキャンセル。ただし unmount 検知が結局必要なので、案1 と組み合わせ。
