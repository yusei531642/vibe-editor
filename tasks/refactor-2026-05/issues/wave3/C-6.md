TITLE: [refactor] pty: window CloseRequested で in-flight inject task を待ってから kill_all する
LABELS: refactor,rust,backend

## 概要 / Summary

vibe-editor のメインウィンドウを閉じる際、`AppHandle::CloseRequested` 同期 callback が即座に `kill_all()` を呼んでおり、`tauri::async_runtime::spawn` 上で進行中の `inject_codex_prompt_to_pty` などの async task を待たずに PTY を kill する。結果、Mutex poison / 半端 inject による不正出力 / reader thread 解放漏れ等の race が起きる。Windows ConPTY では特に顕在化しやすい。

## 根本原因 / Root cause

- `src-tauri/src/lib.rs:269-277`: `WindowEvent::CloseRequested` で `state.terminals.kill_all()` を直呼び。
- in-flight な `tauri::async_runtime::spawn` 内の inject (`inject_codex_prompt_to_pty` 等) が `session.write` 中に kill されると、`SessionHandle::drop` の `killer.lock()` が poison し、`if let Ok(...)` で silently skip される (#C-related: `pty: SessionHandle::drop の Mutex poison`)。
- reader/exit watcher の `std::thread::spawn` も detach されており dispose されない。
- 短命 PTY を多数立てている状態で close すると、orphan thread が数秒間生き続ける。

## 再現手順 / Repro

1. Canvas で agent を 5 枚 recruit、各に inject (team_send) を発火直後にウィンドウ × ボタンで閉じる。
2. プロセス監視 (Process Explorer 等) で vibe-editor 子プロセスが数秒間 zombie で残る場合がある。
3. tracing log を有効化すると "Mutex poison" warning が記録される。

## 影響範囲 / Impact

- Canvas / TeamHub 利用者全員 (close 時の 5-10 秒のもたつき / 子プロセス残留)
- Windows ConPTY ユーザーで顕著
- severity: MEDIUM (close 時のみの race だが、チームプロセスの cleanup 不完全はサポート負担源)

## 修正方針 / Fix proposal

### 案1 (推奨): graceful drain → kill_all のシーケンス化

```rust
WindowEvent::CloseRequested { api, .. } => {
    api.prevent_close();
    let app = app_handle.clone();
    tauri::async_runtime::spawn(async move {
        // 1) inject 系 task の完了を 1 秒以内で待つ (timeout 付)
        let _ = tokio::time::timeout(
            std::time::Duration::from_millis(1000),
            state.terminals.drain_in_flight(),
        ).await;
        // 2) kill_all
        state.terminals.kill_all();
        // 3) exit
        app.exit(0);
    });
}
```

### 案2: SessionHandle に `Arc<AtomicBool> stopping` を持たせ、inject 側がチェック

inject loop の各 chunk write 前に `if session.stopping.load(Ordering::Acquire) { return Err(Stopped); }` をチェック。close 経路で stopping を立てるだけで、async task は自発的に早期 return する。

### 案3: `WindowEvent::Destroyed` を活用

`CloseRequested` ではなく `Destroyed` イベント側で kill_all を呼ぶことで、Tauri 内部の close 処理 (renderer 側 unmount 等) との整合を取る。
