TITLE: [refactor] pty: claude_watcher の deadline を session 寿命に追従させ orphan watcher を減らす
LABELS: refactor,rust,backend

## 概要 / Summary

`claude_watcher` は短命 PTY (起動直後 kill 等) に対しても 60 秒 hard deadline で生き続けるため、30 タブを連続起動 + 即 kill すると `30 × 60` 秒分の watcher thread が同時に並走し、reader thread / channel リソースを長時間専有する。session 寿命に追従するシグナルが無く、`recv_timeout(500ms)` の polling ループに依存しているため反応 lag も大きい。

## 根本原因 / Root cause

- `src-tauri/src/pty/claude_watcher.rs:262-304`
- watcher 内部:
  ```rust
  let deadline = Instant::now() + Duration::from_secs(60);
  loop {
      match rx.recv_timeout(Duration::from_millis(500)) { ... }
      if Instant::now() >= deadline { break; }
  }
  ```
- `is_alive()` の polling 反応 lag が最大 500ms。
- session が 1 秒で kill された場合でも、watcher は 60 秒近く生存する。
- 30 タブ × 60 秒 = 同時並走 watcher thread = 30、reader thread と合わせると Windows のハンドル数が無視できない。

## 再現手順 / Repro

1. Canvas で agent を 30 枚連続 recruit する (各 recruit が短命 PTY を伴う)。
2. 各 agent を 1-2 秒以内に dismiss する。
3. tracing log を有効化すると `claude_watcher` のログが「session は終了済み」状態でも 60 秒近く出続ける。
4. Windows のタスクマネージャーでハンドル数 / スレッド数を観察すると、watcher の遅延に伴うリソース蓄積が見える。

## 影響範囲 / Impact

- Canvas で大量 agent を扱うパワーユーザー
- 短時間に session を入れ替える操作 (HMR / 開発時の頻繁な再起動) を行うユーザー
- リソースリークまでは至らないが、close 時の cleanup を遅らせる
- severity: MEDIUM (refactor / observability)

## 修正方針 / Fix proposal

### 案1 (推奨): is_alive を AtomicBool 化 + crossbeam_channel::select! で multiplex

1. `SessionHandle` に `Arc<AtomicBool> is_alive` を持つ。
2. `claude_watcher` は `crossbeam_channel::select!` で `rx` と「session 終了 signal channel」を multiplex。session が終了した瞬間に watcher が break。
3. deadline (60s) は session-relative (session の start_time + 60s) に変更し、session が終了していたら即座に exit。

### 案2: watcher を tokio::task に書き直し cancel token で停止

`tokio::select!` を使い、`session.cancel_token` の cancellation を検知したら即 drop。tokio の tokio_util CancellationToken を導入。

### 案3 (簡易): polling 間隔を 100ms に短縮 + is_alive 1 回 check で early break

最低限のパッチ。fundamentally は案1 が望ましいが、Wave 3 の手早い緩和としては有効。
