TITLE: [bug] pty: attach 経路で snapshot 〜 listener 登録の窓に emit されたバイトが lost する
LABELS: bug,rust,javascript,backend

## 概要 / Summary

`terminal_create` の `attach_if_exists` 経路で、Rust 側が PTY scrollback の snapshot を作成してから renderer 側が listener を登録するまでの数 ms 〜 数十 ms の窓に emit されたバイトが消える。snapshot 時点の buffer は replay されるが、その後 listener 登録までの間に届いたバイトは「snapshot に含まれない & listener にも届かない」状態になる。

## 根本原因 / Root cause

- `src-tauri/src/commands/terminal.rs:194-205`: snapshot 採取
- `src/renderer/src/lib/hooks/use-xterm-bind.ts:560-630`: queue モードでの listener 登録経路
- snapshot の serialize → IPC return → renderer 側で xterm にデータ流し込み → listener 登録、までの window が 5-50ms 程度。
- この間に PTY から実際にバイトが emit されると、Rust 側の `terminal://output` イベントは飛ぶが、renderer 側ではまだ listener が無いので捨てられる (queue は登録後にしか書き込まれない)。
- Codex Lane 0 / Claude Code 起動時の banner 出力など、attach 直後にバイトが多い場合に欠損が visible になる。

## 再現手順 / Repro

1. Codex 系 PTY を 1 枚立て、すぐに別タブを開いて attach する。
2. Codex の welcome banner / 初期 prompt の一部行が欠落する場合がある。
3. tracing log で `[pty] emit ... bytes=N` と `[renderer] listener registered` のタイムスタンプ差を見ると 10-30ms の窓があり、その間の bytes が renderer に届いていないことが確認できる。

## 影響範囲 / Impact

- attach 経路を使うシナリオ (タブ複製 / Canvas TerminalCard ↔ IDE タブの相互参照 / HMR 後の再 attach)
- 欠損は 1-2 行の文字化けで顕在化することが多く、ユーザーが scrollback を見て初めて気付く
- severity: MEDIUM (UX bug、データロス的だが復元可能なケースが多い)

## 修正方針 / Fix proposal

### 案1 (推奨): snapshot を取る側で side buffer に shadow し、attach 完了 IPC で drain して渡す

1. Rust 側 `terminal_create` (attach 経路) で snapshot を採取した瞬間、その session に対する **side buffer** (Vec<u8>) を `Arc<Mutex<Vec<u8>>>` で起動。
2. 通常の reader thread は emit と並行して side buffer にもバイトを push する。
3. renderer 側が listener 登録を完了したら IPC `terminal_attach_ack` を呼ぶ。
4. Rust 側はその ack を受けて side buffer の中身を drain → 1 回だけ emit → side buffer を破棄。

これで snapshot 〜 listener 登録の間の bytes が確実に renderer に届く。

### 案2: 順序を逆転して listener を先に登録、snapshot を後で取る

renderer が `terminal_create` 前に空の listener を登録 → IPC で snapshot を要求 → snapshot 受信時点ですでに listener が動いている。ただし「listener 登録時点で session ID が分かっていない」ため、session ID を返す IPC を 2 段階に分ける必要があり、refactor が大きい。
